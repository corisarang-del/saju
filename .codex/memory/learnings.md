# Learnings

## 2026-07-24: Paddle webhook은 무서명 차단만으로 충분하지 않다

**확인 내용**:
- 운영 `/api/webhooks/paddle`은 무서명/오서명 payload를 401로 막았다.
- 하지만 로컬 QA 환경의 `PADDLE_WEBHOOK_SECRET`으로 서명한 `transaction.completed`, `transaction.payment_failed`, 중복 `transaction.completed` payload도 운영에서 모두 401로 거부됐다.
- QA 사용자 별 잔액과 거래 로그는 변하지 않았다.

**교훈**:
- webhook 보안 QA는 “401로 막힌다”만 보면 안 되고 정상 서명 성공 경로까지 반드시 확인해야 한다.
- Vercel Production env의 `PADDLE_WEBHOOK_SECRET`과 Paddle dashboard notification destination secret이 같은지 배포마다 확인해야 한다.
- 결제 재오픈 전에는 Paddle simulator 또는 sandbox 실제 결제 payload로 성공/실패/중복 이벤트를 검증해야 한다.
- secret 값은 로그와 문서에 절대 남기지 않고, 존재 여부와 결과 코드만 남긴다.

## 2026-07-24: release gate는 운영 URL QA와 production env 검사를 함께 봐야 한다

**확인 내용**:
- 운영 URL 기준 `pnpm qa:live-api:free`, `pnpm qa:live-api`는 실제 QA 계정과 운영 Supabase를 사용해 통과했다.
- 하지만 `QA_BASE_URL=https://monthlysaju.vercel.app pnpm run release:gate`는 `GOOGLE_VERTEX_RUNTIME_AUTH` production env 검사에서 실패했다.
- 코드 테스트, 타입체크, 린트, 빌드, high 이상 audit은 통과했지만 env 검사 실패 때문에 release gate 전체 통과는 아니다.

**교훈**:
- 운영 API QA 통과와 release gate 전체 통과를 분리해서 기록해야 한다.
- 배포 승인 판정은 production env 검사까지 포함한 `pnpm run release:gate` 기준으로 봐야 한다.
- 결제 포함 재오픈은 별도로 `pnpm run release:gate:payments`까지 통과해야 한다.

## 2026-07-24: IDOR QA는 실제 두 계정과 row unchanged 검증이 필요하다

**확인 내용**:
- 계정 A가 만든 리딩을 계정 B 쿠키로 호출하는 authenticated IDOR QA를 운영에서 수행했다.
- `preview`, `analyze`, `deduct-stars`, `chat`, `pdf`는 404로 막혔다.
- `update-status`는 200을 반환했지만 owner row는 변경되지 않았다.

**교훈**:
- IDOR QA는 응답 코드뿐 아니라 타깃 row가 실제로 변하지 않았는지까지 확인해야 한다.
- no-op 200은 직접 유출/변조가 아니어도 클라이언트와 감사 로그를 혼란스럽게 하므로 404로 정리하는 편이 좋다.

## 2026-07-24: 배포 후 랜딩 디자인 회귀는 실제 뷰포트 확인이 필요하다

**확인 내용**:
- PC 랜딩 캐릭터 캐러셀은 사이드바와 본문 컨테이너 기준이 다르면 첫 카드가 밀리거나 잘린 것처럼 보인다.
- 캐러셀 peek 효과는 그냥 overflow로 잘리는 것보다 `max-w` 본문 정렬, 충분한 gutter, edge fade mask가 함께 있어야 의도처럼 보인다.
- 모바일 390x844에서는 쿠키 안내, 상단 내비게이션, H1, 카드 이미지, 하단 탭이 동시에 높이를 차지하므로 카드 폭과 이미지 비율을 같이 줄여야 첫 CTA가 보인다.
- H1의 외톨이 줄바꿈은 `max-width`와 `font-size`만으로 항상 잡히지 않는다. 한국어 문장은 `inline-block`으로 의미 단위를 묶어 단어 중간 줄바꿈을 막는 편이 안정적이다.
- `/api/analytics/track` 클라이언트 호출이 있으면 운영 완성도를 위해 route를 제공하거나 명시적으로 호출을 꺼야 한다.

**교훈**:
- 디자인 회귀 테스트는 문자열 기준만 두지 말고, Playwright 390x844와 PC 주요 폭 스크린샷까지 함께 봐야 한다.
- 모바일 첫 화면 전환은 CTA가 실제 viewport 안에 보이는지로 판단해야 한다.
- analytics 같은 관측 라우트 404는 사용자 화면을 깨지 않아도 production QA에서는 완성도 이슈로 다룬다.

## 2026-07-10: Gemini 단독 QA 통과와 실제 무료 채팅 API 통과는 다르다

**확인 내용**:
- Gemini 첫 상담 단독 라이브 QA는 6케이스 모두 통과했다.
- 하지만 실제 `/api/saju/chat` 무료 첫 상담 경로는 503으로 실패했다.
- 전체 실사용 QA에서는 Vertex/Gemini 429 자원 소진으로 실패했고, 무료 단독 재시도에서는 3회 모두 품질 게이트를 통과하지 못해 실패했다.
- 유료 첫 상담은 동시 요청에서 200/409 처리, 저장 품질, 별 차감, `chat_used` 증가가 정상으로 확인됐다.
- 후속 개발자 수정 뒤에도 무료 첫 상담 반복 QA에서 200 성공, 503 실패, 품질 미달 저장이 섞여 나왔다.
- 유료 첫 상담도 전체 동시성 QA에서 503/409로 실패한 케이스가 있다.
- 서버 내부 `prevalidated initial analysis generated` 통과 로그가 있어도 외부 QA 기준으로는 `오늘 할 구체 행동 없음` 답변이 저장될 수 있다.

**교훈**:
- LLM 단독 QA 리포트는 실제 API 품질 보증을 대체하지 못한다.
- 무료 경로는 토큰 제한, 재시도 횟수, 품질 게이트 기준이 함께 작동하므로 반드시 실제 API로 회귀 검증해야 한다.
- 3회 재생성 후 실패하는 경로는 저장/차감 방지만으로 충분하지 않고, 사용자에게 응답 실패로 보이는 제품 리스크다.
- 무료 첫 상담은 `isFree: true` 조건에서 별도 안정성 게이트를 둬야 한다.
- 저장 전 게이트와 외부 QA 평가기는 핵심 품질 조건을 공유해야 한다. 특히 `사주 근거`, `오늘 구체 행동`, `질문 종료`, `외래어/영어/이모지 없음`은 같은 판정 로직으로 맞추는 편이 안전하다.

## 2026-07-01: 디자인 개선 재리뷰 결과

**확인 내용**:
- 모바일 하단 쿠키 배너가 입력/CTA를 가리던 P0는 해결됐다.
- 쿠키 배너 개인정보 링크는 i18n 라우팅으로 `/ko/privacy-policy` 흐름을 유지한다.
- 첫 랜딩 캐릭터 이미지 LCP 경고는 `priority`, `loading="eager"`, `fetchPriority="high"` 적용 후 재현되지 않았다.
- 무료 혜택 문구는 `3별 무료`보다 `3회 무료`가 사용자 이해에 좋다.
- 새 디자인 회귀 테스트는 직접 바이너리 기준 2 files / 15 tests 통과했다.
- `pnpm exec vitest ...`는 기존 환경 이슈와 동일하게 `[ERROR] fetch failed`로 실패했다.

**남은 이슈**:
- 모바일에서 쿠키 배너를 상단 fixed overlay로 옮기면서 `/ko` H1과 `/ko/reading` 제목 영역을 덮는다.
- 기능 조작 차단은 아니지만 첫 화면 정보 위계와 제품 첫인상을 해치는 P1로 본다.

**교훈**:
- 하단 입력 차단을 피하려고 상단 overlay로 옮기면 제목/헤더 가림 문제가 생길 수 있다.
- 쿠키 안내는 fixed overlay보다 inline notice 또는 공간 예약형 배너가 모바일 첫 화면에 더 안정적이다.
- 디자인 회귀 테스트는 문자열 검사만으로 충분하지 않고 390x844 시각 겹침 검증이 필요하다.

## 2026-06-30: PM 리뷰 기준 공개 런칭 리스크

**확인 내용**:
- 비공개 베타는 가능하지만 결제 포함 공개 런칭 전에는 보안, 결제, 법무, 빌드 게이트 점검이 필요하다.

- 개인정보 접근 경계: `readingId` 기반 조회는 서버 코드에서도 `user_id` 소유자 조건을 명시하는 편이 안전하다.
- 결제/별 차감: 원자성, 중복 웹훅, 거래 이력 정합성 확인이 필요하다.

- 빌드: Google Fonts fetch 실패 리스크가 있어 로컬 폰트 전환 검토가 필요하다.
- 린트: React 19 규칙 위반이 기존 파일에 남아 있다.

## 2026-07-01: security definer RPC는 grant까지 봐야 한다

**문제**:

**위험**:

- `p_credits`, `p_user_id`, `p_transaction_id`를 인자로 받는 함수는 클라이언트에 열리는 순간 결제 서버 검증을 우회한다.

**교훈**:
- 웹훅 전용 RPC는 `service_role` 전용으로 둬야 한다.

- `authenticated`에 열어둔 별 차감 RPC도 타인 UUID 기반 차감/거래 이력 오염 위험이 있으니 `auth.uid() = p_user_id`와 reading 소유권을 함수 내부에서 강제해야 한다.

## 2026-07-07: 멤버십은 결제 지급과 별도 상태 테이블이 필요하다

**확인 내용**:

**교훈**:

- 구독 상태 upsert 기준은 내부 유저 ID가 아니라 `provider, subscription_id` unique가 더 안전하다.

- memory summary는 최근 사용자 발화만이 아니라 assistant가 이미 안내한 요약을 같이 보관해야 유료 리포트가 같은 말을 반복하지 않는다.

## 2026-07-10: 관리자 수동 별 조정은 감사로그까지 DB RPC로 묶는다

**확인 내용**:
관리자 화면에서 별을 수동 충전/차감할 때 기존 거래 로그만으로는 누가, 왜, 어떤 요청 맥락에서 조정했는지 추적하기 부족했다. 서버 액션에서 balance update와 transaction insert를 따로 실행하면 중간 실패 때 감사 추적이 끊길 수 있다.

**교훈**:
- 관리자 금전성 수동 조정은 actor id/email, target id, before/after, reason, IP, user-agent를 별도 감사로그에 남겨야 한다.
- 잔액 변경, 유저 거래 로그, 관리자 감사로그는 DB RPC 하나로 묶고 row lock을 잡아 원자적으로 처리하는 편이 안전하다.
- 서버 액션의 입력 검증만 믿지 말고 RPC 안에서도 amount, mode, reason을 다시 검증해야 한다.
- 운영 배포 전 Paddle production env 게이트는 로컬 placeholder가 있으면 실패하는 게 정상이고, 실제 운영 값 주입 뒤 다시 실행해야 한다.

- `monthlysaju.vercel.app` production에서 HTTPS redirect, HSTS, 기본 보안 헤더, 비인증 API 차단은 대체로 확인됐다.
- MDN HTTP Observatory는 `B / 75점 / 8개 통과 / 2개 실패`였다.
- `.env`, `.git/config`, `package.json` 공개는 없었고, 공개 HTML에서 명백한 secret 패턴은 발견하지 못했다.
- `/api/saju/chat`, `/api/saju/suggestions`, `/api/webhooks/paddle` 비인증 요청은 401로 막혔다.
- OAuth redirect는 공격자 `origin`/`x-forwarded-host`를 신뢰하지 않았다.
- 반면 OAuth PKCE code verifier cookie는 `Secure`/`HttpOnly` 없이 긴 만료로 내려왔고, CSP에는 `unsafe-inline`, `unsafe-eval`, 넓은 `frame-ancestors`가 남아 있었다.

**교훈**:
- 공개 헤더와 비인증 API가 괜찮아 보여도 IDOR, RLS, 결제 webhook 정합성은 별도 authenticated QA가 필요하다.
- OAuth 관련 쿠키는 인증 성공 전 임시 값이라도 짧은 TTL과 운영 쿠키 속성 기본값을 확인해야 한다.
- `frame-src`와 `frame-ancestors`는 역할이 다르다. 결제 iframe 허용이 필요하다고 해서 외부 사이트가 우리 앱을 감싸는 것을 넓게 허용하면 안 된다.
- Observatory 점수는 헤더 개선 우선순위를 잡는 데 유용하지만, DB 권한/결제 정합성/비즈니스 로직 보안을 보증하지 않는다.

## 2026-07-24: Supabase PKCE 쿠키만 좁게 하드닝한다

**확인 내용**:
Supabase SSR `cookieOptions`를 전역으로 `HttpOnly`나 짧은 TTL로 바꾸면 일반 auth cookie까지 영향을 받아 브라우저 Supabase 클라이언트 세션이 깨질 수 있다. 이번 보안점검의 민감 대상은 `*-auth-token-code-verifier` PKCE 임시 쿠키였으므로 이 이름만 선별해 `HttpOnly`, production `Secure`, `maxAge: 600`으로 좁게 강화했다.

**교훈**:

- 결제 iframe 허용은 `frame-src`, 우리 앱이 iframe으로 감싸지는 권한은 `frame-ancestors`다. Paddle은 `frame-src`에만 남기고 `frame-ancestors`는 기본적으로 `'self'`로 둔다.

- JSON parse 예외는 공통 helper에서 잡고 400으로 내려야 한다.
- 비인증 invalid body는 401 또는 400이어야지 500이 되면 공격자가 5xx 로그와 알람을 쉽게 오염시킬 수 있다.
- 빈 500 body는 클라이언트/모니터링 모두에게 나쁘므로 `{ error, requestId }` 형태의 구조화 응답을 유지한다.

## 2026-07-24: 공격형 점검은 positive QA와 hostile payload QA를 분리한다

**확인 내용**:
운영 live API free/full QA는 통과했고 authenticated IDOR도 막혔다. 하지만 hostile invalid JSON payload에서는 일부 API가 500을 반환했다. 정상 플로우 QA만 보면 놓치는 하드닝 이슈다.

**교훈**:
- release gate에는 정상 사용자 플로우뿐 아니라 invalid content-type, invalid JSON, 악성 origin, forged webhook 같은 hostile payload 회귀를 별도 묶음으로 둔다.
- 성공 경로 통과와 보안 경계 통과를 같은 의미로 보지 않는다.

## 2026-07-24: invalid JSON 방어는 route마다 try/catch를 흩뿌리지 않는다

**확인 내용**:
`preview`, `analyze`, `compatibility`, `chat` route의 invalid JSON 500은 모두 직접 `req.json()`을 호출하는 공통 패턴에서 나왔다. 공통 `safeJson` helper로 묶으니 400 응답 형식, request id 포함, 원문 body 미로그 기준을 테스트로 고정할 수 있었다.

**교훈**:
- 새 API route에서 body parse가 필요하면 `safeJson`부터 쓰고, 직접 `await req.json()`을 새로 추가하지 않는다.
- catch 블록에서 `req.clone().json()`으로 body를 다시 읽는 복구 코드는 invalid body에서 또 실패하므로 요청 초기에 필요한 id만 변수에 보관한다.
- 보안 회귀 테스트는 helper 동작 테스트와 route-level 사용 강제 테스트를 같이 둬야 같은 취약 패턴이 돌아오지 않는다.

## 2026-07-24: Vercel env와 로컬 env 검증은 반영 시점을 분리해서 기록한다

**확인 내용**:
Vercel production env에 `GOOGLE_VERTEX_RUNTIME_AUTH=vercel-oidc`를 추가해도 현재 떠 있는 production 서버에는 다음 배포 전까지 반영되지 않는다. 로컬 `pnpm test:env` 래퍼는 기존 `fetch failed` 문제로 실패했지만, `GOOGLE_VERTEX_RUNTIME_AUTH=vercel-oidc REQUIRE_PRODUCTION_ENV=true node scripts/check-env.js` 직접 실행은 통과했다.

**교훈**:
- Vercel env 변경은 “등록 완료”와 “production 배포 반영 완료”를 따로 추적한다.
- pnpm 래퍼가 fetch failed로 실패하면 동일 검사 스크립트를 직접 실행해 코드/env 스키마 실패와 도구 래퍼 실패를 분리한다.
- Paddle signed QA는 actual product/price id와 webhook secret이 없으면 endpoint 보안 검증까지 갈 수 없다.

## 2026-07-24: 기능 이슈가 닫힌 뒤에는 프리미엄 마감 항목을 따로 티켓화한다

**확인 내용**:
`design-taste-frontend`를 제외하고 `high-end-visual-design`, `minimalist-ui`, `redesign-existing-projects` 기준으로 production 디자인을 다시 보니 릴리즈 차단급 문제는 없었다. 대신 PC 1280x720 CTA 발견성, 캐러셀 첫 진입 예측성, 보라색 accent hierarchy, 카드 표면 물성처럼 프리미엄 마감 이슈가 남았다.

**교훈**:

- “고쳐줘” 문서는 취향 설명보다 완료 기준과 회귀 확인 항목을 넣어야 개발자가 바로 작업할 수 있다.
