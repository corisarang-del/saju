## 2026-07-01: 디자인 개선 재리뷰 결과

**확인 내용**:
- 모바일 하단 쿠키 배너가 입력/CTA를 가리던 P0는 해결됐다.
- 쿠키 배너 개인정보 링크는 i18n 라우팅으로 `/ko/privacy-policy` 흐름을 유지한다.
- 첫 랜딩 캐릭터 이미지 LCP 경고는 `priority`, `loading="eager"`, `fetchPriority="high"` 적용 후 재현되지 않았다.
- 무료 혜택 문구는 `3별 무료`보다 `3회 무료`가 사용자 이해에 좋다.
- 새 디자인 회귀 테스트는 직접 바이너리 기준 2 files / 15 tests 통과했다.
- `pnpm exec vitest ...`는 기존 환경 이슈와 동일하게 `[ERROR] fetch failed`로 실패했다.

**남은 이슈**:
- 모바일에서 쿠키 배너를 상단 fixed overlay로 옮기면서 `/ko` H1과 `/ko/reading` 제목 영역을 덮는다.
- 기능 조작 차단은 아니지만 첫 화면 정보 위계와 제품 첫인상을 해치는 P1로 본다.

**교훈**:
- 하단 입력 차단을 피하려고 상단 overlay로 옮기면 제목/헤더 가림 문제가 생길 수 있다.
- 쿠키 안내는 fixed overlay보다 inline notice 또는 공간 예약형 배너가 모바일 첫 화면에 더 안정적이다.
- 디자인 회귀 테스트는 문자열 검사만으로 충분하지 않고 390x844 시각 겹침 검증이 필요하다.

## 2026-07-01: 랜딩과 입력흐름의 마지막 피로도 조정

**확인 내용**:

- 상담형 앱은 오래 봐도 피곤하지 않게 따뜻한 중립색을 일부 섞는 편이 좋다.

## 2026-07-01: 모바일 쿠키 안내 inline 전환은 최종 재리뷰에서 유효했다

**확인 내용**:

- 문자열 기반 테스트는 최소 방어선이고, 실제 bounding box 겹침 검증을 Playwright로 추가하는 편이 안전하다.

## 2026-07-01: AI 스트림 실패에서 빈 200 응답은 성공보다 나쁘다

**문제**:
Gemini quota exceeded 상태에서 `/api/saju/chat`이 `200 OK`와 빈 body를 반환했고, DB 메시지도 저장되지 않았으며, 서버 로그에는 `[saju/chat] stream error {}`만 남았다.

**교훈**:
- 스트리밍 API는 provider failure를 성공 status로 감추면 사용자가 “답변이 안 나온다”고 느낀다.
- quota/rate-limit 실패는 명확한 UI 메시지와 재시도 안내로 드러내야 한다.
- 로그에는 빈 객체가 아니라 에러 name/message/status/cause/request id를 남겨야 한다.

## 2026-07-02: AI SDK onFinish는 message 단독보다 messages fallback이 안전하다

**문제**:

**확인 내용**:
AI SDK `onFinish`는 `message: this.activeResponse.state.message`와 `messages: this.state.messages`를 함께 넘긴다. 스트림 완료 타이밍이나 내부 상태 반영 방식에 따라 `message.parts`가 비어 있어도 최종 `messages` 배열의 마지막 assistant에는 텍스트가 들어 있을 수 있다.

**교훈**:
- 클라이언트 완료 판정은 `message` 하나만 보지 말고 최종 `messages` 배열의 마지막 assistant를 fallback으로 읽어야 한다.
- 빈 응답 방어문구가 반복되면 provider 실패뿐 아니라 클라이언트 텍스트 추출 오판도 의심해야 한다.
- AI stream 회귀는 서버 저장/차감뿐 아니라 클라이언트 완료 콜백의 상태 판정까지 테스트해야 한다.
- 빌드 산출물 `.next/types/* 4.ts` 같은 중복 파일은 소스 문제가 아니라 타입 체크를 오염시키는 산출물일 수 있으니 구분해야 한다.

## 2026-07-01: security definer RPC는 grant까지 봐야 한다

**문제**:
보안 핫픽스에서 Paddle 웹훅 지급을 원자적 RPC로 옮겼지만, `credit_stars_for_paddle_purchase`가 `security definer`이면서 `authenticated`에 execute grant되어 있었다.

**위험**:
- 로그인 유저가 Supabase RPC로 직접 호출하면 결제 없이 별을 충전할 수 있다.
- `p_credits`, `p_user_id`, `p_transaction_id`를 인자로 받는 함수는 클라이언트에 열리는 순간 결제 서버 검증을 우회한다.
- 기존 `decrement_star`와 `deduct_stars_for_report`도 authenticated 공개 상태면 타인 UUID 기반 차감 DoS 또는 거래 이력 오염 위험이 있다.

**교훈**:
- 웹훅 전용 RPC는 `service_role` 전용으로 둬야 한다.
- 클라이언트 직접 호출 RPC는 함수 내부에서 `auth.uid()`와 소유권을 다시 확인해야 한다.
- 보안 리뷰에서 `security definer`는 함수 본문뿐 아니라 `grant execute`까지 반드시 같이 봐야 한다.
- `authenticated`에 열어둔 별 차감 RPC도 타인 UUID 기반 차감/거래 이력 오염 위험이 있으니 `auth.uid() = p_user_id`와 reading 소유권을 함수 내부에서 강제해야 한다.
