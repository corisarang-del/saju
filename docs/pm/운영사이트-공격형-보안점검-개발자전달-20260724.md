# 운영사이트 공격형 보안점검 개발자 전달

## 요약
- 날짜: 2026-07-24
- 대상: `https://monthlysaju.vercel.app/`
- 요청: 개발자 수정 후 운영 배포본에 대해 직접 공격성 payload를 보내 보안 문제가 있는지 점검
- 방식: 비파괴 공격형 점검. 대량 요청, 실제 사용자 데이터 접근, 서비스 장애 유발 테스트는 하지 않았다.
- 사용 기준: `security-review` 스킬, OWASP IDOR/API1 BOLA, Supabase RLS/RPC, Paddle webhook signature, MDN Observatory
- 소스 수정: 없음

## 최종 판정
- 무료 상담형 운영 핵심 경로는 직접 공격 payload 기준으로 큰 구멍은 발견하지 못했다.
- authenticated IDOR는 이번 수정으로 더 좋아졌다. 이전에 `update-status`가 타인 리딩에 no-op `200`을 주던 문제는 이제 `404`로 바뀌었다.
- 결제 포함 운영/결제 재오픈은 아직 보류해야 한다. Paddle signed webhook 정상 서명 payload가 운영에서 계속 `401`이다.
- production env gate도 `GOOGLE_VERTEX_RUNTIME_AUTH` 때문에 아직 실패한다.
- 새로 발견한 하드닝 이슈는 일부 API가 invalid JSON에 `500`을 반환하는 것이다.

## Findings

### P1. Paddle signed webhook 정상 서명 경로가 여전히 실패

공격/QA 결과:
- 무서명 forged `transaction.completed` payload: `401 Invalid signature`
- 오서명 payload: `401`
- 로컬 QA `PADDLE_WEBHOOK_SECRET`으로 정상 서명한 `transaction.payment_failed`: `401`
- 로컬 QA `PADDLE_WEBHOOK_SECRET`으로 정상 서명한 `transaction.completed`: `401`
- 같은 `transaction.completed` 중복 전송: `401`
- QA 사용자 `user_stars`, `star_transactions`: 변화 없음

해석:
- 웹훅이 완전히 열린 것은 아니다. 위조 payload 차단은 정상이다.
- 하지만 정상 서명 성공 경로도 운영에서 실패하므로, 실제 결제 완료 후 별 지급/멤버십 반영이 실패할 수 있다.
- Vercel Production `PADDLE_WEBHOOK_SECRET`, Paddle Dashboard Notification Destination secret, 로컬 QA secret이 불일치할 가능성이 높다.

개발자 조치:
- Vercel Production env의 `PADDLE_WEBHOOK_SECRET`을 실제 Paddle notification destination secret과 맞춘다.
- Paddle Dashboard webhook endpoint가 `https://monthlysaju.vercel.app/api/webhooks/paddle`인지 확인한다.
- 실제 Paddle simulator 또는 sandbox 결제로 signed payload를 생성해 운영 endpoint에서 확인한다.

수용 기준:
- 무서명/오서명 webhook은 `401`.
- 정상 서명 `transaction.payment_failed`는 `200 ok`이고 별/멤버십 데이터는 변하지 않는다.
- 정상 서명 `transaction.completed`는 `200 ok`이고 실제 paid price id 기준으로 별이 1회만 적립된다.
- 같은 transaction id 중복 전송은 `200 ok`이지만 중복 적립되지 않는다.
- 구독 활성/취소 signed payload는 `user_memberships`에 정확히 반영된다.

### P1. production env gate가 아직 실패

재현:
```bash
REQUIRE_PRODUCTION_ENV=true pnpm test:env
```

결과:
```text
필수 환경변수가 비어 있거나 placeholder 값이야:
- GOOGLE_VERTEX_RUNTIME_AUTH
```

해석:
- 운영 URL live API는 통과하지만, production env 승인 게이트는 아직 통과가 아니다.
- 배포 승인 기준을 release gate 전체 통과로 보면 아직 닫힌 상태다.

개발자 조치:
- 운영 인증 전략에 맞게 `GOOGLE_VERTEX_RUNTIME_AUTH` production env를 실제 허용값으로 등록한다.
- Vercel OIDC/Vertex 인증 전략이면 `scripts/check-env.js`의 허용값과 운영 env 값을 일치시킨다.

수용 기준:
- `REQUIRE_PRODUCTION_ENV=true pnpm test:env` 통과.
- `QA_BASE_URL=https://monthlysaju.vercel.app pnpm run release:gate` 전체 통과.

### P2. invalid JSON/body 처리에서 일부 API가 500 반환

재현:
```bash
curl -s -D - -o - -X POST https://monthlysaju.vercel.app/api/saju/preview \
  -H 'content-type: text/plain' \
  --data 'not-json'

curl -s -D - -o - -X POST https://monthlysaju.vercel.app/api/saju/analyze \
  -H 'content-type: text/plain' \
  --data 'not-json'

curl -s -D - -o - -X POST https://monthlysaju.vercel.app/api/saju/compatibility \
  -H 'content-type: text/plain' \
  --data 'not-json'

curl -s -D - -o - -X POST https://monthlysaju.vercel.app/api/saju/chat \
  -H 'content-type: text/plain' \
  --data 'not-json'
```

실제 응답:
- `/api/saju/preview`: `500 {"error":"Internal server error"}`
- `/api/saju/analyze`: `500 {"error":"Internal server error"}`
- `/api/saju/compatibility`: `500 {"error":"Internal server error"}`
- `/api/saju/chat`: `500` 빈 body

해석:
- 스택 trace나 DB 에러 원문은 노출되지 않았다.
- 다만 잘못된 입력은 `400`, 인증이 필요한 endpoint라면 인증 우선이면 `401`이 더 적절하다.
- 현재 구조는 `req.json()` parse가 인증 확인보다 먼저 실행되거나, JSON parse 예외가 일반 서버 오류로 떨어지는 것으로 보인다.
- 보안상 즉시 침투 성공은 아니지만, 공격자가 invalid body로 5xx 로그를 만들 수 있고 모니터링/알람을 오염시킬 수 있다.

개발자 조치:
- 민감 API는 가능하면 인증 확인을 먼저 수행한 뒤 body parse를 한다.
- body parse는 `try/catch` 또는 공통 helper로 감싸고 JSON 형식 오류는 `400`으로 반환한다.
- `chat` route는 invalid JSON에도 빈 500 body가 아니라 구조화된 JSON error를 반환한다.

수용 기준:
- 비인증 invalid body: `401` 또는 정책상 `400`, 절대 `500` 아님.
- 인증된 사용자 invalid JSON: `400`.
- 응답 body는 `{ error, message?, requestId? }` 형태로 구조화.
- 서버 로그에는 request id와 parse failure만 남기고 stack/원문 body 전체는 남기지 않는다.

### P2. CSP `unsafe-inline` 잔여

현재 상태:
- MDN Observatory 최신 스캔: `B+`, score `80`, failed `1 / 10`
- 실패 항목: CSP `unsafe-inline`
- 개선 확인: `object-src 'none'` 추가됨, `unsafe-eval` 없음, `frame-ancestors 'self'`, COOP/CORP/HSTS 적용

개발자 조치:
- `script-src 'unsafe-inline'`, `style-src 'unsafe-inline'` 제거를 report-only로 먼저 검증한다.
- Next/theme inline script, JSON-LD, Paddle/Google 로딩이 깨지지 않도록 nonce/hash 기반 전환을 검토한다.

수용 기준:
- Observatory CSP 실패 제거 또는 남기는 사유 문서화.
- Google login, Paddle checkout, 채팅, 리포트 페이지가 CSP 강화 후 정상 동작.

## 개발자 작업 순서
1. Vercel Production `PADDLE_WEBHOOK_SECRET`과 Paddle Dashboard notification destination secret 정합성 확인.
2. Paddle simulator/sandbox signed payload로 성공/실패/중복 webhook QA 통과.
3. `GOOGLE_VERTEX_RUNTIME_AUTH` production env 또는 `scripts/check-env.js` 허용값 정합성 수정.
4. invalid JSON/body가 500이 아니라 400/401로 떨어지게 API 공통 body parse/error handling 보강.
5. CSP `unsafe-inline` 제거 또는 report-only 기반 단계적 강화.
6. `QA_BASE_URL=https://monthlysaju.vercel.app pnpm run release:gate` 전체 통과 확인.

## 결론
- 무료 상담형 운영은 직접 공격 payload 기준으로 배포 차단급 신규 취약점은 발견하지 못했다.
- 결제 포함 운영은 아직 차단이다.
- 개발 우선순위는 Paddle signed webhook 정상 경로, production env gate, invalid JSON 500, CSP 순서다.
