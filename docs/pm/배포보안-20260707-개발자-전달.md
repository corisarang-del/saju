## 남은 높은 위험

### P1. 메모리 기반 rate limit/lock은 Vercel 분산 배포에서 전역으로 동작하지 않음

#### 위치
- `src/lib/rate-limit.ts`
- `src/lib/saju/chat-concurrency.ts`
- `src/app/api/saju/chat/route.ts`
- `src/app/api/saju/suggestions/route.ts`

#### 현재 상태
- `chat`과 `suggestions`에 rate limit이 적용됐어.
- `chat`에는 active generation lock도 생겼어.
- 하지만 둘 다 메모리 Map 기반이라 단일 런타임 안에서만 동작해.

#### 위험
- Vercel/서버리스 멀티 인스턴스에서는 요청이 다른 인스턴스로 분산되면 제한이 공유되지 않아.
- 별 1개로 동시 chat 요청이 여러 인스턴스에서 AI 생성까지 갈 수 있어.
- suggestions/chat 비용 폭탄 방어가 운영 트래픽에서 약해질 수 있어.

#### 개발자 조치
- Redis, Upstash, Supabase row lock, DB advisory lock 등 공유 저장소 기반으로 rate limit/lock을 바꿔.
- chat 생성 lock은 최소 `userId` 또는 `userId + readingId` 단위로 전역 보장해야 해.
- 별 차감은 AI 호출 전에 예약/차감하고, 실패 시 환불하는 구조를 검토해.

#### 테스트 제안
- 동일 유저 별 1개 상태에서 동시 요청 2개를 보내 한 개만 AI provider까지 가야 해.
- 분산 환경을 흉내내기 위해 lock store를 공유 저장소 fake로 두고 테스트해.
- rate limit 초과 시 AI provider 호출 전에 `429`가 나와야 해.

### P1. 리포트 생성 실패 시 별 차감 후 stuck 가능성

#### 위치
- `src/app/api/saju/deduct-stars/route.ts`
- `src/app/api/saju/analyze/route.ts`
- `src/components/saju/report/SajuReportClient.tsx`

#### 현재 상태
- 종합 리포트는 서버 RPC에서 5별 차감 후 `saju_readings.status = 'paid'`로 전이해.
- 이후 클라이언트가 `/api/saju/analyze`를 호출해 `paid -> generating -> completed` 흐름을 진행해.

#### 위험
- AI 분석 생성이 실패하면 이미 별은 차감된 뒤야.
- `analyze` catch 블록은 `req.clone().json()`에 기대어 실패 복구를 시도하는데, 요청 body 소비 상태에 따라 복구가 흔들릴 수 있어.
- `generating`에 stuck 되거나 별 차감 후 결과를 못 받는 CS/환불 리스크가 있어.

#### 개발자 조치
- `readingId`를 catch에서 다시 body parse하지 말고 함수 초기에 안전하게 보관해.
- `paid -> generating` 전이를 조건부 update로 처리해 중복 생성 요청을 막아.
- AI 실패 시 `failed` 전이와 재시도/환불/재생성 정책을 명확히 해.
- 차감과 생성 시작 상태를 하나의 서버 오케스트레이션 API로 묶는 방안도 검토해.

#### 수용 기준
- AI provider 실패 후 reading status가 `failed` 또는 재시도 가능한 상태로 안정적으로 남음.
- 별 차감 후 결과 미생성 케이스에 대한 환불/재시도 정책이 있음.
- 같은 reading으로 동시 analyze 요청을 보내도 하나만 생성됨.

### P1. OAuth redirect base URL이 요청 헤더를 우선 신뢰함

#### 위치
- `src/lib/auth/oauth.ts`
- `src/app/api/auth/google/route.ts`
- `src/app/auth/callback/route.ts`

#### 현재 상태
- `Origin`, `x-forwarded-host`, `x-forwarded-proto`를 이용해 OAuth callback/redirect URL을 만들고 있어.
- 터널 개발환경에서는 편리하지만 운영에서는 header 기반 redirect origin 신뢰가 위험해질 수 있어.

#### 개발자 조치
- production에서는 `NEXT_PUBLIC_APP_URL` 또는 별도 `APP_ORIGIN` 고정값을 우선 사용해.
- 터널/로컬 도메인 허용은 development 환경에서만 동작하게 분기해.
- 허용 가능한 host allowlist를 두고, 목록 밖 host는 운영 기본 도메인으로 fallback 해.

#### 수용 기준
- production에서 임의 `Origin`/`x-forwarded-host`로 외부 도메인 callback URL이 만들어지지 않음.
- local/터널 개발환경에서는 기존 테스트 흐름 유지.
