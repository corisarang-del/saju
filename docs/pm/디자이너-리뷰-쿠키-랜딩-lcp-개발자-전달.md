# 디자이너 리뷰 쿠키 랜딩 LCP 개발자 전달

## 배경
- 디자이너/리뷰어 관점에서 모바일 최초 입력 흐름과 랜딩 첫 행동성에 아직 치명적인 마찰이 확인됐다.
- 특히 쿠키 배너가 핵심 입력을 가리는 문제는 단순 카피 문제가 아니라 전환 플로우를 막는 P0 이슈다.
- 이번 전달 문서는 디자이너가 개발자에게 우선순위와 UX 의도를 함께 넘기기 위한 것이다.

## 우선순위 요약
1. P0: 쿠키 배너가 모바일 핵심 입력을 가리는 문제 해결
2. P1: 쿠키 배너 개인정보 링크의 locale 유지
3. P2: 랜딩 첫 캐릭터 이미지 LCP 경고 해결

## P0. 쿠키 배너가 모바일 핵심 입력을 가림

### 발견 위치
- `src/components/ui/cookie-consent.tsx:32`
- `src/components/saju/input/BirthDateForm.tsx:280`

### 문제
- `cookie-consent.tsx`의 fixed 배너가 `/ko/reading` 390x844 모바일 화면에서 `BirthDateForm.tsx`의 달력 구분 영역을 덮는다.
- 다음 버튼은 거의 피했지만, 양력/음력 선택을 막아서 최초 사용자 입력 플로우가 여전히 깨진다.
- 랜딩에서도 카드 하단 `대화하기` CTA가 쿠키 배너와 하단 탭에 가려져 첫 화면 행동성이 약해진다.

### UX 영향
- 신규 사용자는 생년월일 입력 단계에서 “왜 선택이 안 되지?”라고 느낄 수 있다.
- 양력/음력은 사주 계산 신뢰와 직결되는 입력이라 가려지면 치명적이다.
- 쿠키 배너가 신뢰를 주기보다 입력을 방해하는 요소로 느껴질 수 있다.

### 개발 요구사항
- 쿠키 배너가 모바일 입력 영역과 CTA를 덮지 않게 레이아웃을 수정한다.
- `/ko/reading` 390x844에서 양력/음력 선택과 다음 버튼이 모두 클릭 가능해야 한다.
- 랜딩 첫 화면에서 캐릭터 카드 CTA가 쿠키 배너/하단 탭에 가리지 않아야 한다.
- 단순 z-index 조정이 아니라, safe-area, 하단 탭, fixed bottom UI가 함께 있을 때의 실제 사용 가능 영역을 기준으로 해결한다.

### 권장 해결 방향
- 쿠키 배너를 하단 전체 fixed 대신 입력 흐름을 덜 방해하는 위치/크기로 재배치한다.
- 하단 탭이 있는 화면에서는 쿠키 배너 bottom offset을 탭 높이와 safe-area만큼 올리거나, 상단/중앙 compact toast 형태를 검토한다.
- 입력 화면에서는 배너를 더 작은 inline notice 또는 dismissible compact banner로 처리하는 방식도 검토한다.
- CTA와 form control 위에 겹치는 fixed element가 없도록 Playwright 스냅샷으로 확인한다.

### 수용 기준
- `/ko/reading` 390x844에서 양력/음력 선택이 쿠키 배너에 가리지 않는다.
- `/ko/reading` 390x844에서 다음 버튼이 쿠키 배너에 가리지 않는다.
- 랜딩 모바일 첫 화면에서 캐릭터 카드 `대화하기` CTA가 쿠키 배너/하단 탭에 가리지 않는다.
- 쿠키 배너 동의 전/후 모두 입력 플로우가 정상 작동한다.

### QA 체크리스트
- iPhone 390x844
- iPhone 375x812
- Android 360x800
- `/ko/reading` 진입 직후
- 랜딩 첫 화면 캐릭터 카드 영역
- 쿠키 동의 전/후
- 양력/음력 선택 클릭 가능 여부
- 키보드 오픈 상태

## P1. 쿠키 배너 개인정보 링크가 locale을 잃음

### 발견 위치
- `src/components/ui/cookie-consent.tsx:42`

### 문제
- 쿠키 배너 개인정보 링크가 `next/link`로 `/privacy-policy`를 사용하고 있다.
- Playwright 스냅샷에서도 `/privacy-policy`로 잡혔다.
- 현재 locale 페이지 흐름에서는 `/ko/privacy-policy`가 자연스럽다.
- 푸터는 locale 링크가 맞게 잡히는데, 쿠키 배너만 이탈한다.

### UX 영향
- 한국어 페이지에서 쿠키 링크를 눌렀는데 locale 없는 경로로 이동하면 신뢰가 흔들린다.
- 법무/개인정보 링크는 사용자가 민감하게 보는 영역이라 작은 이탈도 신뢰를 낮춘다.

### 개발 요구사항
- 쿠키 배너 개인정보 링크가 현재 locale을 유지하게 수정한다.
- `/ko` 페이지에서는 `/ko/privacy-policy`로 이동해야 한다.
- 다른 locale에서도 해당 locale의 개인정보처리방침 경로로 이동해야 한다.

### 권장 해결 방향
- 프로젝트의 i18n `Link` 래퍼를 사용한다.
- 현재 pathname/locale을 기반으로 locale-aware href를 만든다.
- 푸터와 동일한 라우팅 패턴을 사용해 일관성을 맞춘다.

### 수용 기준
- `/ko` 또는 `/ko/reading`에서 쿠키 개인정보 링크 클릭 시 `/ko/privacy-policy`로 이동한다.
- locale 없는 `/privacy-policy`로 이탈하지 않는다.
- 푸터와 쿠키 배너의 개인정보 링크 동작이 일관된다.

### QA 체크리스트
- `/ko`
- `/ko/reading`
- `/en`, `/ja`, `/zh`가 활성화돼 있다면 각 locale
- 쿠키 동의 전 링크 클릭
- 쿠키 동의 후 배너 미노출 상태

## P2. 캐릭터 카드 첫 이미지 LCP 경고

### 발견 위치
- `src/components/saju/landing/CharacterCards.tsx:121`

### 문제
- 캐릭터 카드를 첫 화면으로 올리면서 첫 캐릭터 이미지가 LCP로 잡힌다.
- 해당 `Image`에 `priority` 또는 eager loading 처리가 없어 Next 경고가 발생했다.
- 랜딩 첫인상 성능과 관련된 경고다.

### UX 영향
- 첫 캐릭터 카드 이미지가 늦게 뜨면 랜딩 첫인상이 흐려진다.
- 20대 여성 고객에게 캐릭터 이미지는 신뢰와 호감의 핵심 신호라 LCP 최적화가 중요하다.

### 개발 요구사항
- 첫 번째 캐릭터 카드 이미지에는 우선 로딩 처리를 적용한다.
- 모든 카드 이미지를 priority로 처리하지 말고, LCP 후보인 첫 카드 중심으로 제한한다.
- carousel/가로 스크롤 구조에서 실제 첫 viewport에 보이는 이미지가 최적화되도록 한다.

### 권장 해결 방향
- `CharacterCard`에 index를 전달한다.
- 첫 카드 또는 현재 첫 viewport에 노출되는 카드에만 `priority`를 적용한다.
- 필요 시 `loading="eager"`와 `fetchPriority="high"` 적용을 검토한다.
- 나머지 카드는 lazy loading을 유지한다.

### 수용 기준
- Next LCP 경고가 사라진다.
- 첫 캐릭터 이미지가 모바일/데스크톱 첫 화면에서 빠르게 표시된다.
- 네트워크 과사용을 막기 위해 모든 카드에 priority가 붙지 않는다.

### QA 체크리스트
- Next dev console 경고 확인
- Lighthouse/LCP 확인
- 모바일 첫 화면
- 데스크톱 첫 화면
- 캐릭터 카드 가로 스크롤

## 디자이너 확인 포인트
- 쿠키 배너는 “안심”을 주되, 입력을 막으면 안 된다.
- fixed 하단 UI가 여러 개 있을 때 실제 터치 가능 영역을 디자인 기준으로 삼는다.
- 개인정보 링크는 법무 링크이므로 locale 이탈 없이 사용자가 보고 있던 언어/흐름을 유지해야 한다.
- 첫 캐릭터 이미지는 랜딩의 감정적 첫인상이므로 LCP 최적화 우선순위가 있다.

## 개발자 전달 시 주의사항
- P0는 카피 개선보다 레이아웃 차단 해소가 핵심이다.
- P1은 라우팅 일관성과 법무 신뢰 문제다.
- P2는 성능 경고지만 전체 이미지 priority 남발은 피한다.
- 구현 전 Playwright 모바일 스냅샷 또는 동등한 재현 테스트를 먼저 잡는다.

## 최종 검증 시나리오
- 쿠키 동의 전 상태로 `/ko/reading` 390x844에 진입한다.
- 양력/음력 선택을 터치한다.
- 이름/생년월일/성별/태어난 시간 입력을 진행한다.
- 다음 버튼을 누른다.
- 랜딩으로 돌아가 캐릭터 카드 CTA가 가리지 않는지 확인한다.
- 쿠키 개인정보 링크가 `/ko/privacy-policy`로 이동하는지 확인한다.
- 랜딩 첫 캐릭터 이미지 LCP 경고가 사라졌는지 확인한다.

## 개발 원칙
- Codex는 PM 역할만 수행한다.
- 소스코드 수정은 개발자가 진행한다.
- 개발자는 TDD 또는 재현 테스트 우선 원칙에 따라 Playwright/스크린샷 검증을 먼저 추가한다.
