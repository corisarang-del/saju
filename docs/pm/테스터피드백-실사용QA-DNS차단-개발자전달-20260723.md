# 테스터 피드백 실사용 QA DNS 차단 개발자 전달 20260723

## 결론

테스터 재검증 결과, 로컬/브라우저/자동 게이트는 대체로 정상이다. 하지만 실제 Supabase DB를 사용하는 API QA는 `sfpwgywcmhgilrqearsz.supabase.co` DNS 해석 실패(`ENOTFOUND`)로 인증 유저 생성 단계에서 중단됐다.

따라서 현재 상태를 “실제 API QA 통과”로 보면 안 된다. 정확한 판정은 **코드 게이트와 화면 렌더링은 통과, 실제 DB 기반 상담/별 차감/거래 로그 QA는 DNS 차단으로 미완료**다.

## 테스터 확인 사항

### 통과

- redirect 정상
- Playwright 렌더링 정상
- 콘솔 에러 없음
- QA 스크립트가 DNS 실패를 구조화해서 보고하는 개선 확인
- Gemini 첫 상담 QA는 통과

### 차단

아래 두 명령이 모두 Supabase DNS lookup 실패로 중단됐다.

```bash
env QA_SCENARIO=free-only node scripts/qa-live-api-check.mjs
node scripts/qa-live-api-check.mjs
```

실패 원인:

- host: `sfpwgywcmhgilrqearsz.supabase.co`
- error: `ENOTFOUND`
- 단계: Supabase auth user 생성 전 DNS preflight/auth 접근
- 결과: 실제 인증 유저 생성까지 도달하지 못함

## 아직 확인 못 한 것

DNS 차단 때문에 아래는 실제 DB 기준으로 끝까지 검증하지 못했다.

- 유료 또는 별 보유 상태 첫 상담 실제 API 응답
- 별 예약/차감
- 실패 시 별 환불
- `star_transactions` 거래 로그
- `chat_used` 증가
- user/assistant 메시지 저장
- 저장된 assistant 응답 품질
- 동시 요청 방어의 실제 DB 반영
- 오늘피드/스냅샷 실제 반영

## 개발자 액션

### P0. Supabase DNS/환경 원인 분리

먼저 앱 로직을 수정하기 전에 DNS/환경 문제를 분리해야 한다.

확인할 것:

1. Supabase project ref `sfpwgywcmhgilrqearsz`가 실제로 살아 있는지
2. `NEXT_PUBLIC_SUPABASE_URL`이 올바른 프로젝트 URL인지
3. 현재 로컬 네트워크에서 해당 host DNS가 막히는지
4. Vercel production env도 같은 잘못된/차단된 Supabase URL을 가리키는지
5. 다른 네트워크 또는 CI/Vercel 환경에서는 DNS가 정상인지

재확인 명령:

```bash
curl -sf -I https://sfpwgywcmhgilrqearsz.supabase.co/auth/v1/health
```

DNS가 계속 실패하면 Supabase project status, project ref, DNS 차단 네트워크 여부부터 확인해.

### P0. 네트워크 정상 환경에서 실제 API QA 재실행

DNS가 정상인 환경에서 아래 두 명령을 다시 실행해야 한다.

```bash
env QA_SCENARIO=free-only node scripts/qa-live-api-check.mjs
node scripts/qa-live-api-check.mjs
```

통과 기준:

- 임시 auth user 생성 성공
- 신규 유저 3회 무료 별 지급 확인
- 무료 첫 상담 응답 저장
- 별 1개 예약/차감
- 실패 시 별 환불
- `star_transactions`에 `chat_message:-1` 저장
- `chat_used` 증가
- user/assistant 메시지 저장
- 동시 요청은 하나만 통과
- 저장된 assistant 품질 게이트 통과

### P0. 배포 판정 문구 정정

개발자/PM/QA 문서에서 이번 상태를 아래처럼 표현해.

- 잘못된 표현: “실제 API QA 통과”
- 올바른 표현: “실제 API QA는 Supabase DNS 차단으로 미완료”
- 올바른 표현: “Gemini QA와 Playwright 렌더링은 통과, 실제 DB 기반 채팅/별 차감 검증은 재실행 필요”

이 상태에서 production 배포 승인을 내면 안 된다. 특히 무료 상담이 실제 DB에서 저장/차감/환불되는지 확인 전에는 “무료 베타 배포 가능”도 최종 승인하면 안 된다.

### P1. QA 스크립트 개선 유지

이번 개선으로 `scripts/qa-live-api-check.mjs`가 DNS 실패를 구조화해서 알려주는 건 좋다.

유지할 기준:

- DNS 실패를 auth 실패나 앱 API 실패처럼 보이게 하지 않는다.
- 실패 응답에 host, error code, next steps가 포함된다.
- DNS preflight 실패 시 이후 destructive/비용 발생 단계로 넘어가지 않는다.

### P1. Gemini QA 통과 후 사람 기준 카피 재검토

Gemini QA는 통과했지만, 관계 고민에서 “몸 상태 기록” 같은 행동 제안이 살짝 어색하게 보였다는 테스터 관찰이 있다.

개발자/프롬프트 담당자는 아래를 확인해.

- 관계/재회 고민에는 몸 상태 기록보다 연락 패턴, 감정 소모, 거리감, 말투, 기다릴 기준 중심 행동을 제안한다.
- 번아웃/건강 고민에는 몸 상태 기록이 자연스럽다.
- 고민 카테고리별 오늘 행동 템플릿이 섞이지 않게 한다.

## 비결제 배포 기준과의 관계

현재 제품 방향은 결제 모듈을 보존하되 production에서는 비활성화하는 무료 상담형 베타다.

Paddle 실제 결제 QA는 이번 배포 범위에서 제외할 수 있다. 하지만 Supabase 기반 무료 상담 QA는 제외하면 안 된다.

즉:

- Paddle env gate 미통과는 이번 비결제 배포 blocker에서 제외 가능
- Supabase DNS/API QA 미완료는 여전히 blocker

## 최종 PM 판단

현재 상태는 “화면과 자동 테스트는 좋아졌지만 실제 DB 기반 실사용 QA가 막힌 상태”다.

개발자는 코드 수정부터 시작하지 말고, 먼저 Supabase DNS/환경을 정상화한 뒤 실사용 API QA를 재실행해야 한다. 그 결과가 통과해야 무료 상담형 베타 배포 판단이 가능하다.
