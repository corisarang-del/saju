## 티켓 1: 쿠키 배너가 핵심 행동을 가리지 않게 수정

### 문제

모바일 `/ko/reading`에서 쿠키 배너가 `다음` 버튼 영역을 덮는다. 랜딩에서도 가격 카드와 캐릭터 CTA를 가린다.

### 원인 후보

- `src/components/ui/cookie-consent.tsx`
- `fixed bottom-4 left-4 right-4 ... z-50`
- 모바일 하단 탭과 폼 버튼을 고려한 safe area/offset이 없음

### 수정 방향

- 모바일에서는 하단 고정 CTA와 탭 바를 피하도록 위치를 조정한다.
- 배너 높이를 줄이고, 필요하면 상단/하단 스택이 아닌 작은 inline consent 형태로 바꾼다.
- `z-50`이 꼭 필요한지 재검토한다. 모달보다 낮고 하단 탭/CTA보다 방해가 덜한 레이어로 정리한다.
- 닫기 버튼은 단순 닫기가 아니라 consent 상태 저장 여부를 명확히 결정한다.

### TDD/검증

먼저 실패하는 검증을 만든다.

- 모바일 390x844에서 `/ko/reading` 진입 시 `다음` 버튼의 bounding box가 쿠키 배너와 겹치지 않아야 한다.
- 모바일 390x844에서 `/ko` 진입 시 캐릭터 카드의 `대화하기` CTA가 쿠키 배너에 가리지 않아야 한다.
- 쿠키 동의/거부/닫기 후 재방문 시 기대한 노출 정책이 유지되어야 한다.

### 수용 기준

- 390x844, 375x667, 430x932에서 제출 버튼과 캐릭터 CTA가 클릭 가능하다.
- 데스크톱에서도 배너가 핵심 카드/가격 정보 위를 덮지 않는다.
- 키보드 포커스가 배너 내부 버튼을 명확히 순회한다.

## 티켓 2: 브랜드 디자인 토큰을 하나로 통합

### 문제

전역 토큰은 다크 퍼플, 랜딩은 베이지/티얼, 입력 폼은 토스 블루다. 사용자는 랜딩, 입력, 채팅, 결제를 이동하며 서로 다른 서비스처럼 느낄 수 있다.

### 대표 위치

- `src/app/globals.css`: 다크 퍼플 토큰
- `src/app/[locale]/page.tsx`: 베이지/티얼 랜딩
- `src/components/saju/input/BirthDateForm.tsx`: `#3182F6` 토스 블루
- `src/components/saju/coin-shop/CoinShopClient.tsx`: 다크 퍼플 결제
- `src/components/saju/layout/MobileLayoutWrapper.tsx`: 모바일 탭 티얼

### 수정 방향

- 브랜드 기본 축을 하나 정한다.
  - 추천: 다크 퍼플을 제품 핵심 경험(채팅/결제)으로 유지하고, 랜딩은 따뜻한 라이트 배경을 쓰되 CTA/선택/과금 단위는 퍼플 계열로 통합.
- `#3182F6` 계열은 제거하거나 브랜드 토큰으로 치환한다.
- 티얼은 보조 성공/안정 신호로만 제한하고, 주 CTA에는 쓰지 않는다.
- 별 과금 단위는 노란 별 + 퍼플 선택 상태처럼 한 조합으로 통일한다.

### TDD/검증

- 새 하드코딩 색상을 추가하지 않는 정적 검사 또는 `rg` 체크를 먼저 둔다.
- 주요 화면에서 CTA 색상 토큰이 동일 계열인지 스냅샷/시각 QA로 확인한다.

### 수용 기준

- `/ko`, `/ko/reading`, `/ko/coin-shop`, `/ko/chat/[characterId]`의 primary CTA가 같은 브랜드 체계로 보인다.
- `#3182F6` 직접 사용이 사라지거나 의도된 토큰으로만 남는다.
- 모바일 탭 active 색상이 랜딩/결제 CTA와 충돌하지 않는다.

## 티켓 3: 사주 정보 입력 컴포넌트 경험 통일

### 문제

`/reading`의 `BirthDateForm`과 채팅 안 `BirthInfoCard`가 다른 디자인과 다른 입력 방식을 쓴다. 한쪽은 이름 단일 입력, 한쪽은 성/이름 분리다.

### 수정 방향

- 공통 입력 컴포넌트 또는 공통 UI 프리셋을 만든다.
- 이름 입력 방식, 년/월/일 표시, 성별/달력 segmented control 스타일을 통일한다.
- 채팅 안 입력은 다크 테마를 유지하더라도 구조와 용어는 `/reading`과 같게 한다.

### TDD/검증

- 같은 데이터 모델을 입력했을 때 두 플로우가 같은 payload를 만든다는 행동 테스트를 먼저 작성한다.
- 년/월/일 유효성, 윤년/월별 일수 처리, 필수값 오류 메시지를 공통 테스트로 둔다.

### 수용 기준

- 사용자는 어느 진입점에서도 같은 필드 순서와 같은 용어를 본다.
- 이름 입력 방식이 하나로 통일된다.
- 오류 메시지가 필드 근처에 일관되게 나온다.

## 티켓 4: 접근성 이름과 키보드 조작 보강

### 문제

Playwright 접근성 스냅샷에서 생년월일 숫자 입력 3개가 이름 없는 `spinbutton`으로 잡혔다. 캐릭터 캐러셀 페이지 점 버튼도 이름이 없다.

### 대표 위치

- `src/components/saju/input/BirthDateForm.tsx`
- `src/components/saju/landing/CharacterCards.tsx`

### 수정 방향

- 년/월/일 input에 `aria-label` 또는 시각적으로 숨긴 label을 연결한다.
- 캐러셀 점 버튼에 `aria-label="n번째 상담사 보기"` 같은 이름을 준다.
- 현재 활성 점에는 `aria-current` 또는 `aria-selected`를 검토한다.
- 캐릭터 카드가 비로그인 상태에서 `div onClick`으로만 동작하지 않게 button semantics를 부여한다.

### TDD/검증

- 접근성 스냅샷에서 세 입력이 각각 `출생 연도`, `출생 월`, `출생 일` 이름을 가져야 한다.
- 캐러셀 점 버튼들이 모두 고유 accessible name을 가져야 한다.
- 키보드만으로 캐릭터 선택, 로그인 모달 열기/닫기가 가능해야 한다.

### 수용 기준

- 이름 없는 interactive element가 없다.
- Tab/Enter/Escape로 주요 흐름이 조작된다.
- 모바일/데스크톱 모두 포커스 링이 보인다.

## 티켓 5: 모바일 랜딩 첫 화면의 행동 약속 정렬

### 문제

H1은 “캐릭터를 고르면”인데 모바일 첫 화면에서는 샘플/가격 카드가 먼저 크게 보이고 캐릭터 선택은 아래로 밀린다.

### 수정 방향

- 모바일에서는 H1 바로 아래에 첫 캐릭터 카드 일부 또는 대표 캐릭터 CTA를 노출한다.
- 가격 카드는 접거나 두 번째 정보 블록으로 낮춘다.
- 첫 화면에서 “캐릭터 선택”과 “무료 3별”이 동시에 이해되게 한다.

### 수용 기준

- 390x844 첫 화면에서 캐릭터 이미지 또는 `대화하기` CTA의 일부가 보인다.
- 가격 정보는 보이더라도 CTA를 밀어내지 않는다.
- 하단 탭과 쿠키 배너가 첫 행동을 방해하지 않는다.

## 티켓 6: 후기 섹션 신뢰도 개선

### 문제

현재 후기 섹션은 “실제 카톡 대화”, “에티오피아 투자까지 맞춤”, 반복적인 `ㅋㅋㅋㅋ`가 강해 조작된 바이럴처럼 보일 수 있다.

### 수정 방향

- 실제 후기라면 출처/동의/익명화 기준을 명확히 한다.
- 너무 극적인 문구보다 구체적인 사용 맥락을 짧게 보여준다.
- “맞춤”처럼 검증 불가능한 단정 표현은 줄인다.

### 수용 기준

- 후기 문구가 과장보다 실제 사용 맥락 중심이다.
- 법무/광고 심의 관점에서 과도한 효능 암시가 줄어든다.
- 20대 여성 고객 기준으로 “재밌지만 수상하다”보다 “써볼 만하다”에 가까워진다.

## 티켓 7: 캐릭터 캐러셀 자동 이동 제어

### 문제

캐릭터 카드가 5초마다 자동 이동한다. 읽는 중 콘텐츠가 움직이고, 모션 감소 설정을 따르지 않는다.

### 수정 방향

- `prefers-reduced-motion`이면 자동 이동을 끈다.
- 사용자가 터치/스크롤/포커스한 뒤에는 자동 이동을 멈춘다.
- 가능하면 자동 이동보다 사용자가 직접 넘기는 구조를 우선한다.

### 수용 기준

- 모션 감소 환경에서 자동 스크롤이 발생하지 않는다.
- 키보드 포커스 중 카드가 움직이지 않는다.
- 모바일에서 읽던 카드가 의도 없이 바뀌지 않는다.

## 티켓 8: 별 과금 단위의 시각 언어 통일

### 문제

별은 과금 단위인데 화면마다 색과 강조 방식이 다르다. 랜딩은 티얼, 결제는 퍼플/노랑, 내비는 앰버가 섞인다.

### 수정 방향

- 별 아이콘, 색상, 배지, 가격 row 표현을 하나의 컴포넌트/패턴으로 통일한다.
- `1별 = 메시지 1회`는 가격 카드, 충전 화면, 페이월에서 같은 톤으로 반복한다.

### 수용 기준

- 별 잔액, 별 충전, 무료 별, 메시지 차감 안내가 같은 시각 언어를 쓴다.
- 과금 관련 CTA는 모두 같은 primary token을 쓴다.
