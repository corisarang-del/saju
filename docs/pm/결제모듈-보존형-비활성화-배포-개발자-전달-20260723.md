# 결제모듈 보존형 비활성화 배포 개발자 전달 20260723

## 결론

이번 배포는 결제 기능을 삭제하거나 구조에서 제거하는 배포가 아니다.

이번 배포의 정확한 목표는 **결제 모듈은 나중에 다시 켤 수 있게 보존하고, production에서는 결제 진입점만 비활성화한 상태로 무료 상담형 베타를 먼저 배포하는 것**이다.

즉, Paddle 관련 타입, 가격 정책, 상품 정책, 별 차감 정책, 웹훅 처리 구조는 유지한다. 다만 Paddle production env와 실제 결제 QA가 아직 준비되지 않았으므로, 유저가 결제 화면에 진입하거나 유료 상품을 구매하려는 흐름은 이번 배포에서 노출하지 않는다.

## 배포 전략

### 이번 배포에서 하는 것

- 무료 상담형 베타 배포
- 가입 후 3회 무료 상담 제공
- 사주 입력, 첫 상담, 오늘피드, 월간 리포트 무료 미리보기 제공
- 별 예약/차감/환불, rate limit, RLS/RPC 보안 유지
- 결제 관련 코드와 정책은 보존
- 결제 진입점은 feature flag로 비활성화

### 이번 배포에서 하지 않는 것

- Paddle 실제 결제 오픈
- 별 충전 구매 오픈
- 월간 멤버십 판매 오픈
- 유료 월간 리포트 상세판 판매 오픈
- 종합 사주 백서 유료 판매 오픈
- Paddle subscription webhook production 운영
- 환불/결제 CS 운영 오픈

## 핵심 원칙

### 1. 결제 코드는 삭제하지 않는다

아래는 나중에 결제 기능을 다시 켤 때 필요하므로 삭제하지 않는다.

- 가격 정책 상수
- 별 상품 타입
- `stars10`, `stars30`, `stars70`, `starsPremium`, `monthlyMembership`
- Paddle config 타입
- Paddle checkout wrapper
- Paddle webhook parsing/verification 구조
- Paddle price id 기반 지급 검증 로직
- 별 차감 정책
- 월간 리포트 3별, 종합 백서 5별 정책
- 결제 관련 테스트

목표는 제거가 아니라 **비활성화 가능한 결제 모듈 보존**이다.

### 2. production에서는 결제 진입점을 닫는다

production 초기 배포에서는 `PAYMENTS_ENABLED=false` 또는 동등한 feature flag를 사용한다.

flag가 false일 때 기대 동작:

- 코인샵 접근 차단 또는 준비 중 화면 노출
- Paddle checkout 호출 차단
- 별 충전 버튼 숨김
- 월간 멤버십 구매 버튼 숨김
- 유료 월간 리포트 상세판 unlock CTA 숨김
- 종합 사주 백서 유료 생성 CTA 숨김
- 결제 완료 redirect, 결제 실패 UI, Paddle customer flow 노출 없음

중요:

- 결제 버튼이 보이는데 클릭하면 실패하는 구조는 안 된다.
- 유저에게 가격을 강하게 노출하고 구매를 기대하게 만들면 안 된다.
- 유료 기능은 “곧 열릴 기능” 정도로 약하게 안내하거나, 아예 숨기는 쪽을 우선한다.

### 3. Paddle env는 이번 배포 필수값에서 제외한다

이번 배포에서는 아래를 release blocker로 보지 않는다.

- `REQUIRE_PADDLE_ENV=true pnpm test:env`
- Paddle client token
- Paddle webhook secret
- Paddle product id
- Paddle price id

대신 기본 env gate는 반드시 통과해야 한다.

- 기본 `pnpm test:env`
- Supabase URL
- Supabase anon key
- Supabase service role key
- AI provider env
- `APP_ORIGIN` 또는 `NEXT_PUBLIC_APP_URL`
- rate limit backend env
- admin allowlist env

### 4. 나중에 결제를 켤 수 있게 경로를 남긴다

결제 재오픈 조건:

1. Paddle production 상품 생성
2. Paddle price id/product id env 등록
3. Paddle webhook secret 등록
4. `PAYMENTS_ENABLED=true`
5. `REQUIRE_PADDLE_ENV=true pnpm test:env` 통과
6. Paddle 실제 결제 QA 통과
7. subscription webhook QA 통과
8. 환불/CS 운영 플로우 준비

이 순서로 결제 기능을 다시 열 수 있어야 한다.

## 개발자 작업 요구사항

### P0. Feature flag 추가 또는 확인

`PAYMENTS_ENABLED` 또는 동등한 flag를 사용한다.

권장 정책:

- local/dev: 필요 시 true 가능
- production 초기 배포: false
- flag 기본값: false

주의:

- env가 없을 때 결제가 켜지면 안 된다.
- Paddle env가 있어도 `PAYMENTS_ENABLED=false`면 결제 UI와 checkout은 막혀야 한다.
- `PAYMENTS_ENABLED=true`인데 Paddle env가 없으면 결제 기능은 fail-closed 해야 한다.

### P0. 결제 진입점 비활성화

확인 대상:

- 랜딩 가격 패널
- 코인샵 페이지
- 채팅 paywall
- 월간 리포트 상세판 unlock
- 종합 사주 백서 생성 CTA
- 멤버십 카드
- Paddle checkout 호출부
- 결제 완료 redirect 처리

요구사항:

- 결제 기능 off 상태에서는 유저가 checkout에 도달할 수 없어야 한다.
- 직접 URL 접근도 막아야 한다.
- 결제 관련 에러가 사용자 첫 상담 흐름을 방해하면 안 된다.

### P0. 무료 상담 플로우 유지

결제를 꺼도 아래 플로우는 살아 있어야 한다.

1. 랜딩 진입
2. Google 로그인
3. 사주 입력
4. 가입 후 3회 무료 별 지급
5. 첫 상담 자동 시작
6. AI 응답 품질 게이트
7. 응답 저장
8. 별 1개 차감
9. 실패 시 별 환불
10. 오늘피드 반영

### P0. Supabase production migration 적용 확인

Paddle과 무관하게 운영 DB에는 아래가 반드시 적용돼야 한다.

- 핵심 개인정보 테이블 RLS
- 별 테이블 직접 조작 차단
- `reserve_chat_star`
- `refund_chat_star`
- 채팅 거래 로그
- rate limit 공유 저장소
- `coaching_snapshots` RLS

확인 기준:

- authenticated 사용자가 `user_stars`, `star_transactions`를 직접 insert/update 할 수 없다.
- 채팅은 AI 호출 전에 별을 예약한다.
- AI 실패/빈 응답/저장 실패/품질 게이트 실패 시 별이 환불된다.
- 거래 로그에 `chat_message:-1`이 남는다.
- RLS disabled advisor 경고가 핵심 테이블에 남지 않는다.

### P0. 실제 API QA 재실행

이전 QA는 Supabase DNS 문제로 실제 API QA가 완료되지 않았다. 네트워크 정상 환경에서 다시 실행해야 한다.

필수 시나리오:

1. 임시 유저 생성
2. 신규 유저 3회 무료 별 지급 확인
3. 무료 첫 상담 요청
4. assistant 응답 저장 확인
5. `chat_used` 증가 확인
6. 별 1개 차감 확인
7. `star_transactions`에 `chat_message:-1` 저장 확인
8. 실패/빈 응답/품질 게이트 실패 시 별 환불 확인
9. 동시 요청은 하나만 통과하고 나머지는 차단되는지 확인
10. 오늘피드가 최신 상담/스냅샷을 반영하는지 확인
11. 결제 feature flag off 상태에서 checkout 직접 접근이 막히는지 확인

이번 QA에서 제외:

- Paddle 실제 결제
- Paddle subscription webhook
- 멤버십 갱신/취소
- 환불 처리

## P1. 배포 화면 정책

### 랜딩 가격 영역

이번 배포에서 가격표를 완전히 지울 필요는 없지만, 결제 가능한 것처럼 보이면 안 된다.

권장 방향:

- 무료 3회 상담을 중심으로 보여준다.
- 가격 정책은 “정식 결제 기능 준비 중” 정도로 낮은 우선순위로 둔다.
- 별 충전/멤버십 구매 CTA는 숨긴다.

권장 문구:

- “가입 후 3회 무료로 먼저 상담해볼 수 있어”
- “유료 충전과 멤버십은 준비 중이야”
- “정식 결제 기능은 안정화 후 열릴 예정이야”

### 월간 리포트

비결제 배포에서는 무료 미리보기 중심으로 둔다.

미리보기에서 강조할 가치:

- 내 고민 기반
- 최근 대화 반영
- 이번 달 선택 3개

유료 상세판은 결제 재오픈 전까지 숨긴다. 무료 별로 상세판을 열게 두면 추후 유료 전환 가치가 흐려질 수 있다.

### 코인샵

권장:

- route 자체를 막거나 준비 중 화면으로 보낸다.
- Paddle checkout은 호출하지 않는다.
- “지금은 무료 상담 베타로 운영 중이야”라는 메시지 정도만 노출한다.

## PM 판단

이번 배포는 결제 기능을 뺀 배포가 아니라 **결제 기능을 보존한 채 production에서 비활성화하는 배포**다.

따라서 개발자는 Paddle 코드를 삭제하지 말고, feature flag로 유저 진입만 닫아야 한다. 무료 상담, 첫 상담 품질, 오늘피드, 월간 리포트 미리보기, 보안/RLS/RPC, 실제 API QA가 통과하면 무료 베타로 배포 가능하다.
