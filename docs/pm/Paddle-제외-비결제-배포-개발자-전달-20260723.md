# Paddle 제외 비결제 배포 개발자 전달 20260723

## 결론

이번 배포는 Paddle 결제/멤버십/유료 충전 기능을 범위에서 제외하고, 무료 상담 중심의 비결제 베타로 배포한다.

중요한 건 Paddle env gate를 그냥 무시하는 게 아니야. 결제 수단이 준비되지 않은 상태에서 가격표, 코인샵, 멤버십, 별 충전, 유료 리포트 CTA가 살아 있으면 유저 신뢰와 CS 리스크가 생겨. 따라서 이번 배포 범위에서는 결제 의존 기능을 숨기거나 “준비 중” 상태로 막고, 무료 3회 상담과 기본 코칭 경험만 안정적으로 열어야 해.

## 이번 배포 범위

### 포함

- 랜딩
- Google 로그인
- 사주 입력
- 가입 후 3회 무료 상담
- 첫 상담 품질 게이트와 안전 fallback
- 오늘피드
- 월간 리포트 무료 미리보기
- 무료 별 잔액 기반 채팅
- 관리자 기본 확인
- Supabase RLS/RPC/별 예약/환불/rate limit 기반 보안

### 제외

- Paddle 결제
- 별 충전 구매
- 월간 멤버십 판매
- Paddle subscription webhook 운영 처리
- 유료 월간 리포트 상세판 판매
- 종합 사주 백서 유료 판매
- 환불/결제 CS 플로우 운영

## P0 출시 전 필수

### 1. Vercel production env 등록 확인

Paddle 관련 env는 이번 배포 blocker에서 제외한다. 대신 아래 env는 반드시 production에 있어야 한다.

- Supabase URL
- Supabase anon key
- Supabase service role key
- AI provider env
- `APP_ORIGIN` 또는 `NEXT_PUBLIC_APP_URL`
- rate limit backend env
- admin allowlist env

요구사항:

- production OAuth origin은 `APP_ORIGIN` 또는 `NEXT_PUBLIC_APP_URL`이 없으면 fail-closed 해야 한다.
- rate limit은 production에서 공유 저장소 기반으로 동작해야 한다.
- Paddle env가 없어도 비결제 모드에서는 build/runtime이 결제 화면으로 유저를 보내면 안 된다.

### 2. Paddle env gate는 이번 배포에서 제외

이번 배포에서는 다음을 release blocker로 보지 않는다.

- `REQUIRE_PADDLE_ENV=true pnpm test:env`
- Paddle product id
- Paddle price id
- Paddle webhook secret
- Paddle client token

대신 개발자는 아래를 구현/확인해야 한다.

- `REQUIRE_PADDLE_ENV`를 켜지 않은 기본 `pnpm test:env`는 통과해야 한다.
- 결제 기능이 꺼진 상태에서 코인샵, 멤버십, 별 충전 CTA, 유료 리포트 unlock CTA가 노출되지 않아야 한다.
- 결제 진입 URL에 직접 접근해도 checkout이 열리지 않고 “결제 기능 준비 중” 또는 홈/상담 흐름으로 안전하게 안내돼야 한다.

권장 구현:

- `PAYMENTS_ENABLED=false` 또는 동등한 feature flag를 둔다.
- production 초기 배포에서는 해당 flag를 false로 둔다.
- flag가 false일 때:
  - 코인샵 진입 차단
  - Paddle checkout 호출 차단
  - 월간 멤버십 카드 숨김
  - 별 충전 버튼 숨김
  - 월간 리포트 유료 상세판 unlock 숨김
  - 종합 사주 백서 유료 생성 CTA 숨김

### 3. Supabase production migration 적용 확인

Paddle 제외와 무관하게 아래는 반드시 운영 DB에 반영돼야 한다.

- RLS/RPC 보안 복구
- `user_stars`, `star_transactions` authenticated 직접 insert/update 차단
- 채팅 별 예약/환불 RPC
- AI 호출 전 별 예약
- 실패/빈 응답/품질 게이트 실패 시 별 환불
- rate limit 공유 저장소 관련 migration
- `coaching_snapshots` RLS

확인 기준:

- authenticated 사용자가 별 테이블을 직접 조작할 수 없다.
- 채팅은 AI 호출 전에 별을 예약한다.
- AI 실패, 빈 응답, 저장 실패, 품질 게이트 실패 시 별이 환불된다.
- `chat_message` 거래 로그가 남는다.
- RLS disabled advisor 경고가 핵심 개인정보 테이블에 남아 있지 않다.

### 4. 실제 API QA 재실행

직전에는 Supabase DNS 실패로 실제 API QA가 미완료였다. 네트워크 정상 환경에서 반드시 다시 실행해야 한다.

필수 시나리오:

1. 임시 유저 생성
2. 신규 가입 3회 무료 별 지급 확인
3. 무료 첫 상담 요청
4. AI 응답 저장 확인
5. `chat_used` 증가 확인
6. 별 1개 차감 확인
7. `star_transactions`에 `chat_message:-1` 저장 확인
8. 실패/빈 응답 상황에서 별 환불 확인
9. 동시 요청에서 하나만 통과하고 나머지는 차단되는지 확인
10. 오늘피드가 최신 상담/스냅샷을 반영하는지 확인

이번 배포에서는 결제/멤버십/Paddle 관련 QA는 제외한다.

## P1 배포 후 보강

### 1. 랜딩 가격 패널 정리

이번 비결제 배포에서는 가격 패널을 그대로 결제 유도처럼 보여주면 안 된다.

권장 문구:

- “가입 후 3회 무료로 먼저 상담해볼 수 있어”
- “유료 충전과 멤버십은 준비 중이야”
- “정식 결제 기능은 안정화 후 열릴 예정이야”

배포 후 결제 재개 시:

- 10별 스타터
- 30/70/250별 패키지
- 월간 멤버십
- 1별 = 메시지 1회
- 월간 리포트 3별
- 종합 백서 5별

이 기준을 다시 가격 패널에 노출한다.

### 2. 월간 리포트 가치 보강

비결제 배포에서는 월간 리포트 무료 미리보기 중심으로 둔다.

개발 요청:

- “내 고민 기반”
- “최근 대화 반영”
- “이번 달 선택 3개”

이 세 가지가 미리보기에서 바로 느껴지게 해.

유료 상세판은 Paddle 재도입 전까지 열지 않는 걸 권장한다. 무료 별 3개로 상세판을 열게 두면 결제 없이 유료 기능 가치가 먼저 소진될 수 있고, 추후 과금 전환이 애매해진다.

### 3. 멤버십 상태 운영 표시는 후순위로 이동

Paddle 제외 배포에서는 멤버십 상태 운영 표시는 release blocker가 아니다.

다만 결제 재도입 전에는 반드시 필요하다.

- 유저별 멤버십 상태
- 최근 결제/갱신/취소
- 최근 별 차감 타입
- 최근 snapshot 생성 여부

## P2 UX/카피 보완

### 1. 어색한 카드 문구 수정

현재 “커리어는 시기야”, “사업은 시기야”는 살짝 부자연스럽다.

권장:

- “커리어는 선택의 흐름이야”
- “사업은 준비와 시기의 균형이야”

### 2. 지안 카드 관계 문구 완화

재회/관계 문구는 불안을 건드리기 쉬워.

개발 방향:

- 겁주는 후킹보다 차분한 상담 톤
- “놓치면 안 돼”류 문구 지양
- “마음 정리와 거리감을 같이 봐줄게”처럼 안정감 있는 표현 사용

### 3. 사이드바 접근성 텍스트 확인

`대화 기록을저장`처럼 붙어 읽힐 수 있는 텍스트를 점검해.

확인 기준:

- 스크린리더/접근성 트리에서 단어가 붙어 읽히지 않는다.
- 버튼/링크 이름이 행동을 분명히 설명한다.
- 모바일 사이드바와 데스크톱 사이드바 모두 확인한다.

## 개발자 작업 순서

1. 결제 feature flag를 확인하거나 추가한다.
2. flag off 상태에서 Paddle 진입점과 유료 CTA를 숨긴다.
3. 기본 env gate와 production origin/rate limit env를 확인한다.
4. Supabase 운영 migration 적용 여부를 확인한다.
5. 실제 API QA를 네트워크 정상 환경에서 재실행한다.
6. 랜딩/리포트/카피/접근성 P2를 다듬는다.
7. 아래 release gate를 통과한 뒤 비결제 베타 배포한다.

## 비결제 배포 release gate

필수 통과:

- `pnpm test`
- `pnpm exec tsc --noEmit`
- `pnpm lint`
- `pnpm build`
- `pnpm audit --prod --audit-level high`
- 기본 `pnpm test:env`
- production Vercel env 확인
- Supabase production migration 적용 확인
- 실제 API QA 통과
- 결제 feature flag off 상태에서 Paddle checkout 접근 불가 확인

이번 배포에서 제외:

- `REQUIRE_PADDLE_ENV=true pnpm test:env`
- Paddle 실제 결제 QA
- Paddle subscription webhook QA
- 환불 처리 QA
- 멤버십 운영 상태 QA

## PM 판단

이번 배포는 “무료 상담형 베타”로는 가능해. 다만 “유료 결제 가능 서비스”로 배포하면 안 된다.

배포 문구와 기능 노출도 이 기준에 맞춰야 해. 유저에게 가격과 결제를 약속하는 화면은 숨기고, 무료 3회 상담과 코칭 품질을 먼저 검증하는 배포로 잡자.
