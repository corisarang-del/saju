# 배포보안 릴리즈게이트 개발자 수정가이드

## 배경
- 배포 전 서버/보안 담당자 관점으로 현재 상태를 다시 점검했어.
- 소스코드는 수정하지 않았고, 배포 차단 가능성이 있는 보안/운영 리스크만 정리했어.
- `pnpm test`, `pnpm lint`, `pnpm build`는 통과했지만 결제 상태 우회와 별 차감 동시성 문제가 남아 있어.

## 결론
- 결제 포함 공개 배포는 아직 보류해야 해.
- 최소한 P0 `update-status` 결제 우회는 배포 전에 반드시 막아야 해.
- 운영 Supabase에는 RPC 권한 회수 마이그레이션을 앱 배포 전에 먼저 적용해야 해.

## 현재 통과한 검증
- `pnpm test`: 통과, 37개 테스트 파일 / 122개 테스트
- `pnpm lint`: 통과
- `pnpm build`: 통과
- `pnpm test:env`: 기본 필수 환경변수 검증 통과

## 현재 실패하거나 운영 확인이 필요한 검증
- `REQUIRE_PADDLE_ENV=true pnpm test:env`: 현재 로컬 환경 기준 실패
  - 누락: `PADDLE_API_KEY`
  - 누락: `PADDLE_WEBHOOK_SECRET`
  - 누락: `NEXT_PUBLIC_PADDLE_CLIENT_TOKEN`
  - 누락: `NEXT_PUBLIC_PADDLE_PRODUCT_STAR_30`
  - 누락: `NEXT_PUBLIC_PADDLE_PRICE_STAR_30`
  - 누락: `NEXT_PUBLIC_PADDLE_PRODUCT_STAR_70`
  - 누락: `NEXT_PUBLIC_PADDLE_PRICE_STAR_70`
  - 누락: `NEXT_PUBLIC_PADDLE_PRODUCT_STAR_PREMIUM`
  - 누락: `NEXT_PUBLIC_PADDLE_PRICE_STAR_PREMIUM`
- `pnpm audit --prod`: 실패
  - moderate: `postcss < 8.5.10`
  - low: `@ai-sdk/provider-utils <= 3.0.97`

## 이전 보안 리뷰에서 개선된 내용
- 핵심 사주 데이터 접근에 인증/소유자 검증이 추가됐어.
- `/api/saju/deduct-stars`는 클라이언트 `amount`를 직접 믿지 않고 서버 정책/RPC 기반으로 바뀐 것으로 보여.
- Paddle 웹훅은 브라우저 `customData.productType`보다 실제 결제 `items.price.id` 기반 지급으로 개선됐어.
- `decrement_star`, `deduct_stars_for_report`, `credit_stars_for_paddle_purchase`는 새 마이그레이션에서 `authenticated` 실행 권한을 회수하고 `service_role` 중심으로 정리됐어.
- `/api/saju/suggestions`는 인증, 입력 길이 제한, user/IP rate limit, daily quota, provider 실패 처리까지 들어갔어.

## P0. `update-status` 결제 상태 우회

### 위치
- `src/app/api/saju/update-status/route.ts`
- `src/app/api/saju/analyze/route.ts`

### 문제
- `update-status`는 인증과 소유자 검증은 하지만 클라이언트가 보낸 `status` 값을 그대로 저장해.
- 사용자가 자기 reading의 상태를 `paid`로 바꾸면 `analyze`에서 유료 분석 조건을 통과할 수 있어.
- 결제나 별 차감이 실제로 성공했는지와 무관하게 유료 분석이 열릴 수 있는 결제 우회 경로야.

### 수정 요구사항
- 일반 클라이언트 요청으로 `paid` 상태를 직접 만들 수 없게 해.
- `paid` 전이는 결제 성공 또는 별 차감 성공이 확인된 서버 내부 경로에서만 허용해.
- 허용 가능한 상태 전이를 enum으로 제한해.
- 현재 상태와 다음 상태 조합을 검증해.
- 잘못된 전이는 `400` 또는 `403`으로 실패시키고 DB 상태가 바뀌지 않아야 해.

### 테스트 제안
- 실패 테스트를 먼저 작성해.
- 로그인 사용자 A가 자기 reading에 `status: "paid"`를 직접 보내도 실패해야 해.
- 실패 후 해당 reading의 status가 바뀌지 않아야 해.
- 정상 결제/별 차감 서버 경로에서는 `paid` 전이가 가능해야 해.
- `analyze`는 단순 `status === "paid"`만 믿지 않고 결제/차감 성공 상태와 연결된 서버 신뢰 경계를 확인해야 해.

### 수용 기준
- 클라이언트가 `/api/saju/update-status`로 임의 `paid` 상태를 만들 수 없어.
- 결제/차감 실패 상태에서 `/api/saju/analyze`가 실행되지 않아.
- 상태 전이 정책이 테스트로 고정돼 있어.

## P1. 운영 DB 마이그레이션 선적용 필요

### 위치
- `supabase/migrations/202607010020_rpc_suggestions_security.sql`

### 문제
- 코드 저장소에는 RPC 보안 마이그레이션이 있지만, 운영 Supabase에 실제 적용됐는지는 로컬 코드만으로 보장할 수 없어.
- 이 마이그레이션이 빠진 상태로 앱만 배포하면 과거 `authenticated` RPC 실행 권한이 남아 있을 수 있어.

### 배포 요구사항
- 앱 배포 전에 운영 Supabase에 `202607010020_rpc_suggestions_security.sql` 적용 여부를 확인해.
- 운영 DB에서 민감 RPC 권한을 직접 조회해 `public`, `anon`, `authenticated` 실행 권한이 없는지 확인해.
- `service_role` 경로에서만 별 충전/차감 RPC가 호출되는지 확인해.

### 테스트 제안
- 사용자 A 세션으로 `decrement_star`, `deduct_stars_for_report`, `credit_stars_for_paddle_purchase` 직접 호출을 시도해 실패하는지 확인해.
- 실패 후 별 잔액과 거래 이력이 변하지 않아야 해.
- 서버 API 경로에서는 정상 케이스가 유지돼야 해.

### 수용 기준
- 운영 DB 기준 민감 RPC는 클라이언트 권한으로 직접 실행할 수 없어.
- 마이그레이션 적용 순서가 배포 체크리스트에 들어가 있어.

## P1. 채팅 AI 생성 동시 요청과 별 차감 레이스

### 위치
- `src/app/api/saju/chat/route.ts`

### 문제
- 별 잔액 확인은 AI 생성 전에 하고, 별 차감은 스트림 완료 후에 해.
- 별 1개 상태에서 동시 요청을 여러 개 보내면 둘 다 사전 잔액 확인을 통과할 수 있어.
- 그 뒤 한 요청만 차감되고 다른 요청은 차감 실패하더라도 이미 AI 비용과 응답 저장이 발생할 수 있어.

### 수정 요구사항
- 사용자/reading 단위 active generation lock을 둬서 동시에 여러 생성이 시작되지 않게 해.
- 가능하면 별 차감 예약 또는 원자적 차감이 AI 호출 전에 성공해야 생성이 시작되게 해.
- 실패한 AI 생성은 별을 환불하거나 차감 예약을 취소하는 보상 흐름을 명확히 해.
- 분당 rate limit과 daily quota를 chat route에도 적용해.
- 차감 실패는 단순 로그로 끝내지 말고 사용자 응답, 저장 여부, 과금 상태를 일관되게 처리해.

### 테스트 제안
- 별 1개인 동일 사용자로 동시 채팅 요청 2개를 보내 하나만 성공해야 해.
- 실패한 요청은 assistant 메시지 저장과 별 차감이 일어나지 않아야 해.
- AI provider 실패 시 별이 차감되지 않거나 예약 차감이 환불돼야 해.
- rate limit 초과 시 AI provider가 호출되기 전에 `429`가 나와야 해.

### 수용 기준
- 별 1개로 유료 AI 응답 2개를 받을 수 없어.
- 동시 요청 실패 상태에서도 DB 메시지, `chat_used`, 별 잔액이 서로 맞아.
- abuse 상황에서 AI 비용이 먼저 발생하지 않아.

## P1. Paddle 운영 환경변수 확인

### 위치
- `scripts/check-env.js`
- `.env.example`

### 문제
- 기본 `pnpm test:env`는 통과하지만 Paddle 검증은 `REQUIRE_PADDLE_ENV=true`일 때만 실행돼.
- 운영 배포에서 Paddle 값이 빠져도 기본 검증만 보면 통과로 오해할 수 있어.
- `.env.example`에는 `NEXT_PUBLIC_PADDLE_ENVIRONMENT=sandbox`, `NEXT_PUBLIC_APP_URL=http://localhost:3000` 예시가 있어 운영 값과 혼동하면 결제가 깨질 수 있어.

### 배포 요구사항
- 결제 포함 운영 배포에서는 `REQUIRE_PADDLE_ENV=true pnpm test:env`를 배포 전 게이트로 실행해.
- 운영 Vercel 환경변수에 Paddle production 값이 들어갔는지 확인해.
- `NEXT_PUBLIC_APP_URL`은 운영 도메인으로 설정해.
- Paddle webhook endpoint와 secret이 운영 앱 기준으로 연결돼 있는지 확인해.

### 수용 기준
- 운영 배포 전 Paddle env 검증이 통과해.
- sandbox 값으로 production 배포하지 않아.
- 실제 Paddle 결제 1건 또는 sandbox와 분리된 운영 사전 검증 절차가 있어.

## P2. 의존성 취약점

### 위치
- `pnpm audit --prod`

### 문제
- `postcss < 8.5.10` moderate 취약점이 있어.
- `@ai-sdk/provider-utils <= 3.0.97` low 취약점이 있어.

### 수정 요구사항

- lockfile 업데이트 후 `pnpm audit --prod`, `pnpm test`, `pnpm build`를 다시 돌려.

### 수용 기준
- production audit에서 최소 moderate 이상 취약점이 없어.
- 패치 후 빌드와 핵심 테스트가 통과해.

## P2. 운영 AI 인증과 CSP 하드닝

### AI 운영 인증
- Vertex 모드는 빌드만으로 실제 인증 성공을 보장하지 않아.
- Vercel 운영 환경에는 `AI_PROVIDER`, `AI_MODEL`, `GOOGLE_VERTEX_PROJECT`, `GOOGLE_VERTEX_LOCATION`과 런타임 인증 방식이 정확히 있어야 해.
- 운영에서 실제 `/api/saju/chat` 1회 호출로 AI 응답, 저장, 별 차감까지 확인해야 해.

### CSP
- `next.config.ts`에는 보안 헤더가 있지만 CSP에 `unsafe-inline`, `unsafe-eval`이 남아 있어.
- Next/Paddle 때문에 필요할 수 있지만 XSS 방어력은 약해져.
- 운영 안정화 후 nonce/hash 기반으로 줄일 수 있는지 검토해.

## 배포 체크리스트
- P0 `update-status` 결제 우회 테스트가 실패에서 성공으로 바뀌었는지 확인해.
- 운영 Supabase에 RPC 권한 회수 마이그레이션을 먼저 적용해.
- 민감 RPC 직접 호출이 클라이언트 권한에서 실패하는지 확인해.
- 채팅 동시 요청에서 별 1개로 응답 2개가 생성되지 않는지 확인해.
- `REQUIRE_PADDLE_ENV=true pnpm test:env`를 운영 환경 기준으로 통과시켜.
- `pnpm audit --prod`에서 production 취약점을 정리해.
- 운영 AI 호출, 메시지 저장, 별 차감, `chat_used` 증가를 실제로 확인해.
- `pnpm test`, `pnpm lint`, `pnpm build`를 최종 통과시켜.

## 개발 원칙
- TDD로 실패 테스트를 먼저 작성해.
- 보안 테스트는 내부 함수 호출 여부보다 실제 관찰 가능한 상태를 검증해.
- 결제/별/AI 비용 경계는 클라이언트 입력을 신뢰하지 말고 서버의 원자적 상태 전이로 닫아야 해.
