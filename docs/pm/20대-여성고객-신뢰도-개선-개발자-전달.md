# 20대 여성고객 신뢰도 개선 개발자 전달

## 배경
- 20대 여성 유저 관점에서 현재 서비스는 “한 번 써볼 만한 사주앱” 수준까지 올라왔다.
- 다음 전환율과 신뢰도를 올리기 위해서는 입력 직전 개인정보 안심, 후기 과장감 완화, 약관/다국어 복붙 흔적 제거가 가장 중요하다.
- 목표 체감 점수는 현재보다 한 단계 높은 8.5 수준이다.

## 우선순위 요약
1. 생년월일 입력 직전 개인정보 안심 문구 추가
2. 년/월/일 입력 접근성 개선
3. 후기 과장 문구 완화
4. `messages/*.json`의 이전 SaaS 문구 제거
5. 캐릭터 원본 프롬프트 말투 정리

## 티켓 1. 생년월일 입력 전 개인정보 안심 문구 추가

### 문제
- 이름, 생년월일, 성별을 입력하기 전 화면이 아직 건조하다.
- 민감한 개인정보를 넣기 직전에 “왜 필요한지, 어떻게 쓰이는지, 삭제 가능한지”가 보이지 않아 이탈 가능성이 있다.

### 요구사항
- 생년월일/이름/성별 입력 화면 상단 또는 입력 영역 바로 위에 짧은 안심 문구를 추가한다.
- 문구는 과하게 법무 문서처럼 쓰지 말고, 20대 여성이 읽었을 때 부드럽고 명확해야 한다.

### 권장 카피
- `입력한 정보는 사주 분석에만 사용돼. 원하면 언제든 삭제 요청할 수 있어.`
- `생년월일과 성별은 사주 계산에만 쓰고, 광고나 다른 목적으로 사용하지 않아.`
- `처음이라 조심스러울 수 있어서, 분석에 필요한 정보만 받을게.`

### 수용 기준
- 로그인 전/후 생년월일 입력 화면에서 안심 문구가 보인다.
- 모바일 화면에서 문구가 입력 필드를 밀어내거나 버튼을 가리지 않는다.
- 문구에 “삭제 요청 가능” 또는 동등한 의미가 포함된다.

### QA 체크리스트
- 모바일 375px 폭에서 문구 줄바꿈이 자연스러운가
- 입력 폼 진입 직후 바로 보이는가
- 개인정보처리방침 링크가 있다면 정상 이동하는가
- 말투가 서비스 전체 톤과 맞는가

## 티켓 2. 년/월/일 입력 접근성 개선

### 문제
- 년/월/일 입력칸이 숫자칸 3개처럼 보여서 순간적으로 “어디가 월이지?”라고 헷갈릴 수 있다.
- 특히 모바일에서 빠르게 입력하는 사용자는 placeholder만으로는 부족할 수 있다.

### 요구사항
- 각 입력칸 위 또는 내부 주변에 `년`, `월`, `일` 라벨을 명확히 표시한다.
- 라벨은 스크린리더 접근성에도 연결되어야 한다.
- 플레이스홀더만 의존하지 않는다.

### 권장 UI 방향
- 세 칸 각각에 상단 라벨: `태어난 연도`, `월`, `일`
- 입력칸 오른쪽 단위 텍스트: `년`, `월`, `일`
- 예시 보조문구: `예: 1998 / 03 / 21`

### 수용 기준
- 각 필드가 시각적으로 무엇을 의미하는지 즉시 구분된다.
- `label` 또는 `aria-label`이 각 입력과 연결된다.
- 모바일에서 숫자 키보드 사용이 유지된다.

### QA 체크리스트
- iOS/Android 모바일에서 숫자 키보드가 뜨는가
- 탭 순서가 연도 → 월 → 일 순서인가
- 빈 값/오류 상태에서 어떤 칸이 문제인지 알 수 있는가

## 티켓 3. 후기 과장감 완화

### 문제
- 후기의 `에티오피아 투자까지 맞춤` 문구는 20대 여성 유저에게 신기함보다 광고 냄새로 느껴질 수 있다.
- 사주앱의 신뢰도는 “말도 안 되게 맞춘다”보다 “내 상황을 부드럽게 짚는다” 쪽이 더 설득력 있다.

### 요구사항
- 과장되거나 너무 특이한 성공담 느낌의 후기를 줄인다.
- 후기 톤은 친구에게 공유할 법한 자연스러운 말투로 바꾼다.

### 권장 카피 방향
- `요즘 답장 하나에도 너무 흔들렸는데, 일단 내 기준부터 보라는 말이 좋았어.`
- `이직 고민 중이었는데 무조건 옮기라는 말이 아니라 지금 체크할 조건을 정리해줘서 편했어.`
- `사주앱인데 생각보다 말투가 부담스럽지 않아서 계속 물어보게 됐어.`

### 수용 기준
- 후기에서 과도한 투자/수익/성공 확정처럼 보이는 표현이 제거된다.
- 후기 카피가 실제 20대 여성 사용자의 말투처럼 자연스럽다.
- 후기가 서비스 신뢰를 해치지 않는다.

## 티켓 4. `messages/*.json` 이전 SaaS 복붙 흔적 제거

### 문제
- `messages/*.json`에 SDS, 화학, LemonSqueezy, `your-email@example.com` 같은 이전 SaaS 문구가 남아 있다.
- 약관/개인정보/다국어 화면에 노출되면 신뢰가 크게 깨질 수 있다.

### 요구사항
- `messages/ko.json`, `messages/en.json`, `messages/ja.json`, `messages/zh.json` 전체에서 이전 SaaS 문구를 검색하고 제거한다.
- 결제 제공자는 현재 정책에 맞게 Paddle로 통일한다.
- 이메일은 `corisarang@gmail.com` 또는 프로젝트에서 확정한 지원 이메일로 통일한다.
- 화학/SDS/규제 문구는 월간사주 맥락의 사주/채팅/결제/개인정보 문구로 대체한다.

### 반드시 검색할 키워드
- `SDS`
- `화학`
- `chemical`
- `LemonSqueezy`
- `your-email@example.com`
- `CAS`
- `GHS`
- `안전보건자료`

### 수용 기준
- `messages/*.json`에서 위 키워드가 남아 있지 않다.
- 다국어 약관/개인정보 화면이 월간사주 서비스 맥락과 일치한다.
- 한국어 기준 문구와 영어/일본어/중국어 문구의 정책 내용이 충돌하지 않는다.

### QA 체크리스트
- 각 locale에서 약관 페이지 확인
- 각 locale에서 개인정보처리방침 확인
- 검색엔진 구조화 데이터와 화면 문구의 가격/이메일/결제사가 일치하는지 확인

## 티켓 5. 캐릭터 원본 프롬프트 말투 정리

### 문제
- 현재 화면 노출 단계에서는 말투가 정제됐지만, 캐릭터 원본 프롬프트에는 `니 사주`, `형이` 같은 표현이 남아 있다.
- 치환 구조에 의존하면 AI 답변 톤이 재발할 수 있다.

### 요구사항
- 원본 캐릭터 프롬프트 자체에서 하대감 있는 표현을 제거한다.
- 치환 함수는 보조 안전장치로만 남기고, 원본 데이터도 안전한 톤으로 만든다.
- 직설 캐릭터의 개성은 유지하되 무례함, 비꼼, 겁주기 톤은 줄인다.

### 말투 기준
- `니` → `네`
- `형이 봐줄게` → `내가 봐줄게`
- `어이 동생` → `반가워`
- `당하는 거야` → `놓치기 쉬워`
- 경고 후에는 반드시 실행 가능한 행동 제안

### 수용 기준
- 원본 프롬프트 검색 시 `니 사주`, `형이`, `어이 동생`이 남아 있지 않다.
- 테스트 또는 스크립트로 금지 표현 회귀를 막는다.
- 실제 AI 첫 답변 샘플에서 무례한 표현이 재발하지 않는다.

## 최종 QA 시나리오
- 신규 유저가 랜딩에서 무료 맛보기 결과와 가격을 확인한다.
- 캐릭터를 고른다.
- 생년월일 입력 직전에 개인정보 안심 문구를 본다.
- 년/월/일을 헷갈리지 않고 입력한다.
- 고민 선택지에서 생활 언어 라벨을 본다.
- AI 첫 답변이 직설적이어도 무례하지 않다.
- 푸터, 약관, 개인정보, 다국어 문구에서 이전 SaaS 흔적이 보이지 않는다.
