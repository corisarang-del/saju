# 보안리뷰 P0 개인정보 결제 핫픽스 개발자 전달

## 배경
- 보안담당자 리뷰에서 공개 출시를 막아야 하는 P0 보안 이슈가 발견됐다.
- 이번 리뷰는 `security-review` 스킬 기준으로 진행됐고, 소스코드는 수정하지 않았다.
- 핵심 위험은 개인정보 IDOR, 별 무단 충전, 결제 검증 우회다.

## 결론
- 결제 포함 공개 배포는 P0 해결 전까지 보류해야 한다.
- 임시 UX 개선보다 RLS 복구, 소유자 검증, 별 차감 원자화, Paddle 지급 검증이 우선이다.
- 개발자는 TDD 방식으로 재현 테스트를 먼저 만들고, 핫픽스 구현 후 회귀 테스트를 통과시켜야 한다.

## 우선순위 요약
1. P0: 핵심 사주 데이터 RLS 복구와 모든 `readingId`/`compatibilityId` 소유자 검증
2. P0: 별 차감 API의 클라이언트 `amount` 신뢰 제거와 원자적 차감
3. P0: Paddle 웹훅 지급 기준을 실제 결제 `items.price.id`로 변경하고 idempotency 보장
4. P1: AI 비용 발생 API 인증, rate limit, daily quota 적용
5. P1: Server Action의 `readingId` 기반 소유자 검증 추가
6. P2: 관리자 감사 로그/IP 제한/추가 인증, OAuth redirect origin 고정

## P0. IDOR / 개인정보 노출

### 발견 위치
- `supabase/migrations/202606300010_core_saju_schema.sql:149`
- `supabase/migrations/202606300030_disable_core_saju_rls.sql:7`
- `src/app/api/saju/analyze/route.ts:27`
- `src/app/api/saju/chat/route.ts:73`
- `src/app/api/saju/pdf/[id]/route.ts:21`
- `src/services/saju/actions.ts:203`

### 문제
- 핵심 사주 테이블의 RLS가 꺼져 있다.
- `preview`, `analyze`, `compatibility`, `chat`, 결과 페이지, PDF API가 `readingId`만으로 조회하고 `user_id` 소유자 검증을 하지 않는다.
- 다른 사람의 생년월일, 이름, 분석 결과, PDF에 접근할 가능성이 있다.

### 개발 요구사항
- 핵심 사주 테이블 RLS를 복구한다.
- 모든 `readingId`/`compatibilityId` 조회에 `user_id = current user id` 조건을 추가한다.
- 서버 API, Server Action, 결과 페이지, PDF API가 같은 권한 정책을 쓰게 정리한다.
- 존재하지 않는 ID와 타인 소유 ID는 정보 노출이 적은 응답으로 처리한다.

### 수용 기준
- 로그인 사용자 A가 사용자 B의 `readingId`로 analyze/chat/PDF/result/getReading에 접근하면 403 또는 404가 반환된다.
- RLS가 켜진 상태에서도 정상 사용자는 자신의 리딩을 생성, 조회, 분석, 채팅, PDF 생성할 수 있다.
- `compatibilityId`가 있는 모든 흐름도 소유자 검증을 통과해야만 접근 가능하다.
- 회귀 테스트에 cross-user access 시나리오가 포함된다.

## P0. 별 무단 충전

### 발견 위치
- `src/app/api/saju/deduct-stars/route.ts:5`
- `src/app/api/saju/deduct-stars/route.ts:26`

### 문제
- `/api/saju/deduct-stars`가 클라이언트가 보낸 `amount`를 그대로 믿는다.
- 음수 `amount`를 보내면 `newBalance = balance - amount` 계산 때문에 잔액이 늘어날 수 있다.
- 조회 후 업데이트 방식이라 동시 요청 race condition에도 약하다.

### 개발 요구사항
- 클라이언트가 별 차감 수량을 직접 결정하지 못하게 한다.
- 서버 고정 가격표 또는 사용 행위별 비용 정책으로 차감량을 결정한다.
- 별 차감은 PostgreSQL RPC나 트랜잭션으로 원자적으로 처리한다.
- 잔액 부족, 음수/0 입력, 동시 요청에 대한 회귀 테스트를 추가한다.

### 수용 기준
- 음수 `amount` 요청은 잔액을 늘리지 못하고 거절된다.
- 클라이언트가 임의의 `amount`를 보내도 서버 정책 외 차감/충전이 발생하지 않는다.
- 동시 요청으로 잔액이 음수가 되거나 중복 사용되는 상황이 막힌다.
- 별 차감과 거래 기록이 같은 원자적 흐름으로 남는다.

## P0. Paddle 결제 검증 우회

### 발견 위치
- `src/lib/paddle/client.ts:122`
- `src/app/api/webhooks/paddle/route.ts:33`
- `src/app/api/webhooks/paddle/route.ts:76`
- `supabase/migrations/202606260050_star_credit_schema.sql:10`

### 문제
- Paddle 웹훅은 서명은 검증하지만, 지급 크레딧을 실제 결제된 `priceId/items/amount`가 아니라 브라우저가 넣은 `customData.productType`으로 정한다.
- 싼 상품 결제 후 `customData.productType=starsPremium`으로 조작하면 250개 지급 가능성이 있다.
- `paddle_transaction_id` unique/idempotency가 없어 중복 웹훅 재처리 위험이 있다.

### 개발 요구사항
- 웹훅 지급 기준을 Paddle 이벤트의 실제 `items.price.id` 또는 서버에 저장된 checkout/session 정보로 변경한다.
- `customData`는 사용자 매핑 등 보조 정보로만 쓰고, 상품/지급량의 신뢰 근거로 쓰지 않는다.
- `paddle_transaction_id` 또는 Paddle event/transaction 고유값에 unique constraint를 추가한다.
- 중복 웹훅은 idempotent하게 무시되도록 처리한다.

### 수용 기준
- 싼 상품 결제 이벤트에 `customData.productType=starsPremium`이 들어와도 실제 결제 상품 기준으로만 별이 지급된다.
- 동일 Paddle transaction/event가 두 번 들어와도 별이 중복 지급되지 않는다.
- 알 수 없는 `price.id`는 별 지급 없이 보안 로그/오류로 처리된다.
- 결제 성공, 상품 불일치, 중복 웹훅 회귀 테스트가 있다.

## P1. AI 비용 폭탄

### 발견 위치
- `src/lib/rate-limit.ts:29`
- `src/app/api/saju/suggestions/route.ts:4`
- `src/lib/saju/ai/analyzer.ts:59`

### 문제
- 비용 발생 API에 rate limit과 daily quota가 붙어 있지 않다.
- `checkRateLimit` 유틸은 있지만 실제 API 사용처가 없다.
- 특히 `suggestions`는 인증 없이 Gemini 호출이 가능하다.

### 개발 요구사항
- AI 호출 API에 인증 또는 엄격한 IP/user rate limit을 적용한다.
- 무료 사용자, 로그인 사용자, 결제 사용자별 daily quota를 정의한다.
- `suggestions`는 인증 없이 호출 가능하게 둘 경우 IP 기반 제한과 abuse 방어를 붙인다.

### 수용 기준
- 비인증 사용자가 `suggestions`를 무제한 호출할 수 없다.
- 동일 사용자/IP가 quota를 넘기면 429가 반환된다.
- 정상 사용자의 핵심 분석 흐름은 quota 정책 안에서 유지된다.

## P1. Server Action 권한 누락

### 발견 위치
- `src/services/saju/chat-actions.ts:37`
- `src/services/saju/chat-actions.ts:58`
- `src/services/saju/chat-actions.ts:119`
- `src/services/saju/chat-actions.ts:150`

### 문제
- `updateReadingMeta`, `getChatMessages`, `getChatCredits`, `addChatCredits`가 `readingId`만 받고 소유자 확인을 하지 않는다.
- RLS가 꺼진 상태라 Server Action 경유로도 IDOR이 발생할 수 있다.

### 개발 요구사항
- Server Action도 API route와 동일하게 현재 사용자와 리딩 소유자를 검증한다.
- 읽기/쓰기/크레딧 변경 액션마다 권한 정책을 명확히 분리한다.

### 수용 기준
- 사용자 A가 사용자 B의 `readingId`로 chat messages, credits, metadata를 조회/수정할 수 없다.
- 권한 실패 시 민감 정보가 응답에 포함되지 않는다.

## P2. 관리자 하드닝 부족

### 발견 위치
- `src/utils/auth/adminGuard.ts:37`
- `src/services/admin/stars.ts:186`

### 문제
- 관리자 권한이 이메일 allowlist에만 의존한다.
- 별 수동 조정 같은 민감 작업에 감사 로그, IP allowlist, 별도 관리자 인증 레이어가 부족하다.
- 현재 트랜잭션 기록은 남지만 관리자 요청 메타데이터 수준의 감사 로그는 부족하다.

### 개발 요구사항
- 별 수동 조정에는 관리자 ID, 대상 사용자, 변경 전/후, 사유, IP, user-agent, request id를 남긴다.
- 운영 환경에서는 IP allowlist 또는 추가 관리자 인증 레이어를 검토한다.
- 관리자 액션은 최소 권한 원칙으로 나눈다.

### 수용 기준
- 별 수동 조정 이벤트를 누가, 언제, 왜, 어디서 실행했는지 추적할 수 있다.
- 일반 allowlist 관리자와 결제/크레딧 관리자 권한을 분리할 수 있는 설계가 잡힌다.

## P2. OAuth redirect hardening

### 발견 위치
- `src/lib/auth/oauth.ts:4`
- `src/app/api/auth/google/route.ts:11`

### 문제
- 로그인 시작 시 `Origin` 헤더를 우선 신뢰해서 callback URL을 만든다.
- Supabase redirect allowlist가 엄격하면 막히겠지만, 서버에서는 고정 origin을 우선 쓰는 편이 안전하다.

### 개발 요구사항
- callback origin은 `NEXT_PUBLIC_APP_URL` 또는 서버 환경변수의 고정 origin을 우선 사용한다.
- 헤더 기반 origin은 개발 환경 등 제한된 조건에서만 fallback으로 사용한다.

### 수용 기준
- 운영 환경에서 임의 Origin 헤더로 OAuth callback URL이 바뀌지 않는다.
- Supabase redirect allowlist와 앱 서버 callback 생성 규칙이 일치한다.

## 검증 메모
- `pnpm audit --prod` 결과 `postcss <8.5.10` moderate 1건이 확인됐다.
- 경로는 `next@16.2.9 > postcss@8.4.31`이다.
- 시크릿 문자열 스캔에서는 실제 키로 보이는 값은 발견되지 않았고, `llm.md`에 예시 prefix만 있었다.

## 개발자 테스트 계획
- 보안 회귀 테스트는 구현 전 먼저 작성한다.
- IDOR: 사용자 A/B를 만들고 A가 B의 `readingId`/`compatibilityId`로 API, Server Action, PDF, 결과 조회를 시도하는 테스트를 추가한다.
- 별 차감: 음수 amount, 0 amount, 임의 amount, 잔액 부족, 동시 요청 테스트를 추가한다.
- Paddle: 실제 `items.price.id` 기준 지급, `customData` 변조, 중복 웹훅 테스트를 추가한다.
- AI 비용: 비인증 호출, quota 초과, 정상 호출 테스트를 추가한다.
- OAuth: 운영 환경에서 임의 Origin 헤더가 callback URL에 반영되지 않는 테스트를 추가한다.

## 릴리즈 게이트
- P0 3건이 해결되기 전에는 결제 포함 공개 출시를 진행하지 않는다.
- P1은 공개 출시 전까지 해결해야 한다.
- P2는 출시 전 하드닝 작업으로 잡되, 관리자 별 수동 조정은 운영 접근 권한이 있는 기능이라 가능하면 P1 일정에 포함한다.

## 개발 원칙
- Codex는 PM 역할만 수행한다.
- 소스코드 수정은 개발자가 진행한다.
- 개발자는 TDD 방식으로 재현 테스트를 먼저 만들고 구현한다.
- 테스트는 구현 세부가 아니라 관찰 가능한 보안 동작을 검증한다.
