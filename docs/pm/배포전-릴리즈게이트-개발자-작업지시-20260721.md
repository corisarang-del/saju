# 배포 전 릴리즈게이트 개발자 작업지시

## 목적
- Vercel `monthlysaju` production 배포 전에 결제, 시크릿, 별 차감, RLS, AI 비용, rate limit, 의존성 취약점 리스크를 닫는다.
- 이 문서는 개발자가 바로 티켓으로 나눠 처리할 수 있는 작업지시다.
- 모든 구현은 TDD로 진행한다. 먼저 실패 테스트를 만들고, 통과 구현 후 `pnpm` 검증 게이트를 다시 돈다.

## P0 릴리즈 차단 작업

### 1. Vercel production env 전체 등록
**작업**
- Vercel 프로젝트 `monthlysaju` production 환경에 필요한 env를 전부 등록한다.
- 등록 대상:
  - Supabase URL, anon key, service role key.
  - AI provider 관련 env.
  - Vertex/Gemini provider env.
  - Paddle API key, webhook secret, client token.
  - Paddle product id, price id.
  - `APP_ORIGIN` 또는 `NEXT_PUBLIC_APP_URL`.
  - `ADMIN_EMAILS`.
  - `REQUIRE_PADDLE_ENV=true`.

**주의**
- service role, Paddle API key, webhook secret은 절대 `NEXT_PUBLIC_`로 만들지 않는다.
- production에서 placeholder, test key, 빈 문자열이 있으면 배포 실패로 처리한다.
- 실제 키값은 문서나 git에 남기지 않는다.

**수용 기준**
- `REQUIRE_PADDLE_ENV=true pnpm test:env`가 production env 기준으로 통과한다.
- Vercel production build에서 Paddle 필수 env 누락이 있으면 실패한다.
- `NEXT_PUBLIC_` env에는 공개 가능한 값만 있다.

### 2. Supabase 별 테이블 RLS 정책 잠금
**작업**
- Supabase에서 `user_stars` 직접 insert/update 정책을 제거한다.
- Supabase에서 `star_transactions` 직접 insert 정책을 제거한다.
- 별 지급/차감/조정은 service role 또는 security definer RPC만 가능하게 한다.

**필수 테스트**
- authenticated 유저가 `user_stars`를 직접 insert/update 할 수 없음을 확인한다.
- authenticated 유저가 `star_transactions`를 직접 insert 할 수 없음을 확인한다.
- service role/RPC 경로로만 별 차감과 거래 로그가 함께 생성되는지 확인한다.

**수용 기준**
- 클라이언트 anon/authenticated key로 별 잔액 조작이 불가능하다.
- 모든 별 변동은 `star_transactions`에 남는다.
- 관리자 수동 조정은 감사로그까지 남는다.

### 3. 채팅 별 차감 원자성 재설계
**문제**
- 현재 채팅은 AI 응답 생성과 저장, 별 차감이 분리돼 있다.
- AI 호출은 성공했는데 차감이 실패하거나, 응답 저장은 됐는데 거래 로그가 누락되는 상태를 막아야 한다.

**작업 옵션**
- 옵션 A: AI 호출 전 별 1개를 예약/차감하고, AI 생성 또는 저장 실패 시 환불 RPC를 호출한다.
- 옵션 B: DB 트랜잭션형 RPC로 assistant 저장, `chat_used` 증가, 별 차감, 거래 로그 생성을 한 번에 처리한다.

**권장 방향**
- 장기적으로는 옵션 B가 더 안전하다.
- 구현 난도가 높으면 v1 배포 전에는 옵션 A로 먼저 막고, 환불 로그까지 남긴다.

**필수 테스트**
- 별이 0개면 AI provider가 호출되지 않는다.
- AI 응답 저장 실패 시 별 차감이 남지 않거나 환불 로그가 남는다.
- 별 차감 RPC 실패 시 assistant 응답 저장과 `chat_used` 증가가 실패 처리된다.
- 동시 요청에서 별 1개로 AI 응답 2개를 받을 수 없다.

**수용 기준**
- 응답 저장, `chat_used`, 별 차감, 거래 로그가 서로 불일치하지 않는다.
- 실패 케이스는 200으로 덮지 않고 4xx/5xx로 명확히 반환한다.

### 4. rate limit 공유 저장소 전환
**문제**
- in-memory rate limit은 서버리스/다중 인스턴스에서 우회된다.

**작업**
- Upstash Redis, Vercel KV, Supabase RPC 중 하나로 rate limit 저장소를 교체한다.
- 최소 기준:
  - userId + IP 기준 분당 제한.
  - userId 기준 일일 제한.
  - AI 비용 발생 API 별도 제한.
  - 관리자 계정 우회 정책은 별도 audit log와 함께 둔다.

**필수 테스트**
- 같은 userId/IP가 제한을 넘으면 429를 받는다.
- 서버 인스턴스가 달라도 같은 제한이 적용된다.
- 제한 초과 시 AI provider가 호출되지 않는다.

**수용 기준**
- `/api/saju/chat`, `/api/saju/analyze`, 결제/리포트 차감 API가 공유 저장소 기반 제한을 사용한다.
- 제한 실패는 사용자에게 재시도 가능한 메시지와 429 상태로 반환된다.

### 5. Paddle subscription webhook 검증 강화
**작업**
- one-time credit 지급처럼 subscription webhook도 product id, price id를 서버 allowlist로 검증한다.
- Paddle event id 또는 subscription id 기반 idempotency를 보장한다.
- `subscription.activated`, `subscription.updated`, `subscription.canceled` 모두 상태 전이를 검증한다.

**필수 테스트**
- 잘못된 price id/product id subscription event는 멤버십을 만들거나 갱신하지 않는다.
- 같은 event를 두 번 받아도 멤버십/별 지급이 중복되지 않는다.
- webhook 서명 검증 전에는 DB 변경이 일어나지 않는다.

**수용 기준**
- Paddle dashboard에서 잘못 연결된 상품이나 공격성 payload로 멤버십 권한이 생기지 않는다.
- 테스트 키와 실키 혼용이 production에서 막힌다.

### 6. 의존성 취약점 정리
**작업**
- `pnpm audit --prod`를 실행한다.
- production 취약 패키지를 업데이트한다.
- major update가 필요한 경우 변경 영향 범위를 기록한다.

**필수 테스트**
- 업데이트 후 `pnpm test`, `pnpm exec tsc --noEmit`, `pnpm lint`, `pnpm build`를 통과한다.

**수용 기준**
- production high/critical 취약점이 남아 있으면 배포하지 않는다.
- 해결 불가한 취약점은 영향도, exploit 가능성, 임시 완화책을 릴리즈 노트에 남긴다.

## P1 배포 안정화 작업

### 7. production release gate 스크립트 묶기
**작업**
- 배포 전 한 번에 실행할 스크립트를 추가한다.
- 예시:
  - `REQUIRE_PADDLE_ENV=true pnpm test:env`
  - `pnpm test`
  - `pnpm exec tsc --noEmit`
  - `pnpm lint`
  - `pnpm build`
  - `pnpm audit --prod`

**수용 기준**
- 개발자가 배포 전 하나의 명령으로 production gate를 확인할 수 있다.

### 8. 운영 QA 문서 업데이트
**작업**
- 실제 production URL 기준 QA 체크리스트를 만든다.
- 체크 항목:
  - Google OAuth.
  - 첫 상담 품질.
  - 무료 3회와 별 차감.
  - 월간 리포트 3별.
  - 종합 백서 5별.
  - Paddle 결제 성공/중복 webhook.
  - 관리자 별 조정 감사로그.

**수용 기준**
- QA가 개발자 도움 없이 production smoke test를 반복할 수 있다.

## 배포 금지 조건
- Vercel production env 중 필수 값이 비어 있거나 placeholder다.
- `SUPABASE_SERVICE_ROLE_KEY`나 Paddle secret이 `NEXT_PUBLIC_`로 노출됐다.
- authenticated 유저가 별 잔액이나 거래 로그를 직접 조작할 수 있다.
- AI 호출 성공 후 별 차감 실패가 성공 응답으로 처리된다.
- in-memory rate limit만 남아 있다.
- Paddle subscription event가 price/product id 검증 없이 멤버십을 만든다.
- `pnpm audit --prod` high/critical 취약점이 설명 없이 남아 있다.

## 최종 검증 명령
```bash
REQUIRE_PADDLE_ENV=true pnpm test:env
pnpm test
pnpm exec tsc --noEmit
pnpm lint
pnpm build
pnpm audit --prod
```
