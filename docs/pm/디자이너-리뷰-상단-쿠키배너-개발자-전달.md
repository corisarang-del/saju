# 디자이너 리뷰 상단 쿠키배너 개발자 전달

## 배경
- 이전에 P0였던 하단 쿠키 배너의 입력/CTA 차단 문제는 해결됐다.
- 개인정보 링크 locale 유지와 첫 캐릭터 이미지 LCP 경고도 개선됐다.
- 다만 쿠키 배너가 상단 fixed 형태로 바뀌면서 모바일 첫 화면 제목을 덮는 P1 이슈가 남았다.

## 개선 확인
- 쿠키 배너가 달력 선택과 다음 버튼을 더 이상 막지 않는다.
- 개인정보 링크가 `/ko/privacy-policy`로 이동해 locale을 유지한다.
- 첫 캐릭터 이미지 LCP 경고가 사라졌다.
- `src/components/saju/landing/CharacterCards.tsx:135`의 `priority`, `eager`, `fetchPriority` 적용이 먹힌 것으로 보인다.
- `3별 무료`보다 `3회 무료` 중심으로 정리되어 무료 혜택이 훨씬 직관적이다.
- 접근성 라벨, 가격 배지 분리, 후기 톤도 좋아졌다.

## 남은 이슈 요약
1. P1: 상단 fixed 쿠키 배너가 모바일 `/ko` H1 일부와 `/ko/reading` 제목 영역을 가린다.
2. P2: 모바일 첫 화면에서 쿠키 안내가 정보 위계를 깨지 않도록 최종 시각 QA가 필요하다.

## P1. 상단 쿠키 배너가 모바일 첫 화면 제목을 덮음

### 발견 위치
- `src/components/ui/cookie-consent.tsx:32`

### 문제
- 상단 fixed 쿠키 배너가 모바일 `/ko`의 H1 일부를 가린다.
- `/ko/reading`에서도 제목 영역을 덮어 첫 입력 화면의 정보 위계가 깨진다.
- 이전 P0였던 하단 입력/CTA 차단은 해결됐지만, fixed overlay 방식 자체가 모바일 첫 화면과 충돌하고 있다.

### UX 영향
- 신규 사용자는 첫 화면에서 핵심 메시지를 완전히 읽기 전에 쿠키 배너를 먼저 보게 된다.
- 입력 화면에서는 제목과 맥락 설명이 가려져 “무엇을 입력하는 화면인지” 순간적으로 흐려질 수 있다.
- 쿠키 배너가 신뢰 요소가 아니라 화면을 방해하는 요소처럼 느껴질 수 있다.

### 개발 요구사항
- 모바일에서는 쿠키 배너가 H1, 페이지 제목, 핵심 CTA, 입력 컨트롤을 덮지 않게 수정한다.
- fixed overlay보다 inline notice, 공간 예약형 배너, 또는 콘텐츠 흐름 안의 compact notice를 우선 검토한다.
- 상단 fixed를 유지해야 한다면 페이지 최상단에 배너 높이만큼 공간을 예약해서 제목이 가려지지 않게 한다.
- `/ko`와 `/ko/reading` 모두에서 쿠키 동의 전/후 상태를 확인한다.

### 권장 해결 방향
- 랜딩: hero/H1 위나 아래에 inline cookie notice로 배치하고, 콘텐츠를 밀어내는 방식으로 처리한다.
- 입력 화면: 제목 아래 또는 폼 하단의 non-overlay notice로 두되, 입력 컨트롤을 덮지 않는다.
- 전역 fixed overlay를 유지할 경우 safe-area와 header offset뿐 아니라 실제 콘텐츠 시작 위치까지 계산한다.
- 쿠키 동의 후 레이아웃 점프가 과하지 않게 처리한다.

### 수용 기준
- 모바일 `/ko` 첫 화면에서 H1이 쿠키 배너에 가리지 않는다.
- 모바일 `/ko/reading`에서 제목 영역이 쿠키 배너에 가리지 않는다.
- 달력 선택, 다음 버튼, 캐릭터 CTA는 계속 가리지 않는다.
- 개인정보 링크는 계속 `/ko/privacy-policy`로 locale을 유지한다.
- 첫 캐릭터 이미지 LCP 경고가 다시 발생하지 않는다.

### QA 체크리스트
- 360x800, 375x812, 390x844 모바일 viewport
- `/ko` 쿠키 동의 전 첫 화면
- `/ko/reading` 쿠키 동의 전 첫 화면
- 쿠키 동의 후 레이아웃 점프
- H1/제목 가림 여부
- 달력 선택/다음 버튼 터치 가능 여부
- 랜딩 캐릭터 CTA 터치 가능 여부
- 개인정보 링크 locale 유지
- Next dev console LCP 경고 재발 여부

## 개발자 전달 시 주의사항

- 단순 z-index 조정보다 overlay를 콘텐츠 흐름 안으로 넣는 방식이 모바일 UX에 더 맞다.
- Codex는 PM 역할만 수행한다. 소스코드 수정은 개발자가 진행한다.
