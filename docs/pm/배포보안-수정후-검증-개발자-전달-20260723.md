## 수정 확인된 보안 항목

### 1. Supabase 별 직접 조작 차단
확인 위치:
- `supabase/migrations/202607210010_release_gate_star_reservation_rate_limit.sql`

확인 내용:
- `user_stars`의 기존 authenticated insert/update 정책 제거
- `star_transactions`의 기존 authenticated insert 정책 제거
- 별 예약/환불은 `reserve_chat_star`, `refund_chat_star` RPC 중심으로 이동
- rate limit은 `rate_limits`, `check_rate_limit` RPC로 공유 저장소 기반 방어를 구성
- 민감 RPC execute 권한은 `service_role` 중심으로 제한

남은 조건:
- 이 마이그레이션이 production Supabase DB에 실제 적용됐는지 별도로 확인해야 해.
- 코드 저장소에 migration이 있어도 운영 DB 적용 여부는 보장되지 않아.

### 2. 채팅 별 차감 원자성 개선
확인 위치:
- `src/app/api/saju/chat/route.ts`

확인 내용:
- AI 호출 전 `reserve_chat_star`로 별 1개를 예약 차감함
- 실패 시 `refund_chat_star`로 환불함
- stream error, incomplete assistant response, assistant persistence failure, chat generation failure 경로에 환불 처리가 들어감
- 동시 요청 회귀 테스트가 통과함

남은 조건:
- production DB에 새 RPC가 적용돼 있어야 실제 운영에서도 동작해.

### 3. Paddle 멤버십 webhook 검증 개선
확인 위치:
- `src/lib/paddle/membership.ts`
- `src/lib/paddle/membership.test.ts`

확인 내용:
- webhook item의 `price.id`를 필수로 검증함
- 설정된 Paddle product/price allowlist와 매칭함
- `product_id`와 `price_id` 불일치를 거부함
- 월간 멤버십이 아닌 price가 subscription webhook으로 들어오면 거부함

### 4. production rate limit fail-closed
확인 위치:
- `src/lib/rate-limit.ts`

확인 내용:
- `RATE_LIMIT_BACKEND=supabase`면 Supabase RPC `check_rate_limit` 사용
- production에서 shared backend가 아니면 `RATE_LIMIT_SHARED_BACKEND_REQUIRED`로 실패함

남은 조건:
- Vercel production env에 `RATE_LIMIT_BACKEND=supabase`를 반드시 넣어야 해.
- 안 넣으면 안전하게 막히긴 하지만, 운영 AI API가 정상 동작하지 않아.

### 5. release gate 강화

### P0. Next.js high 취약점으로 production audit 실패
검증 명령:
```bash

```

결과:
- `next@16.2.9`에서 high 취약점 4개 확인
- 패치 버전: `next >=16.2.11`

취약점:
- Next.js Middleware / Proxy bypass in App Router applications using Turbopack and single locale
- Next.js Denial of Service in App Router using Server Actions
- Next.js Server-Side Request Forgery in Server Actions on custom servers
- Next.js Server-Side Request Forgery in rewrites via attacker-controlled destination hostname

개발자 조치:
- `next`를 `>=16.2.11`로 올려.
- lockfile 갱신 후 아래를 다시 돌려.

재검증:
```bash
pnpm test
pnpm exec tsc --noEmit
pnpm lint
pnpm build
pnpm audit --prod --audit-level high
```

수용 기준:
- high 이상 production 취약점이 없어.
- Next 업데이트 후 OAuth, middleware/proxy, rewrite, server action 관련 회귀가 없어.

## 운영 적용 확인이 필요한 항목

### Supabase migration 적용 확인
필수 확인:
- `202607210010_release_gate_star_reservation_rate_limit.sql`가 production DB에 적용돼야 해.
- `reserve_chat_star`, `refund_chat_star`, `check_rate_limit` 함수가 production DB에 있어야 해.
- `user_stars`, `star_transactions`에 authenticated 직접 insert/update 정책이 남아 있으면 안 돼.

권장 확인:
```sql
select policyname, cmd
from pg_policies
where schemaname = 'public'
  and tablename in ('user_stars', 'star_transactions');
```

```sql
select routine_name
from information_schema.routines
where routine_schema = 'public'
  and routine_name in ('reserve_chat_star', 'refund_chat_star', 'check_rate_limit');
```

### OAuth origin 설정
확인 위치:
- `src/lib/auth/oauth.ts`
- `src/lib/auth/oauth.test.ts`

현재 상태:
- production에서 `APP_ORIGIN` 또는 `NEXT_PUBLIC_APP_URL`이 있으면 untrusted forwarded header보다 configured origin을 우선함
- 하지만 production env가 비어 있으면 header fallback 경로가 남아 있어

개발자 조치:
- Vercel production에 `APP_ORIGIN=https://운영도메인` 또는 `NEXT_PUBLIC_APP_URL=https://운영도메인`을 반드시 넣어.
- 가능하면 production에서 canonical origin 누락 시 fail-closed 하도록 추가 하드닝을 검토해.

## 개발자 후속 처리 결과

### 처리 완료
- `next`를 `^16.2.11`로 업데이트해 문서에서 지적한 Next.js high 취약점 패치 기준을 충족했어.
- `eslint-config-next`는 `16.2.11`, `sharp`는 `^0.35.0`으로 맞췄어.
- `pnpm-lock.yaml`을 갱신했고 로컬 `node_modules`도 lockfile 기준으로 재설치했어.
- production에서 `APP_ORIGIN` 또는 `NEXT_PUBLIC_APP_URL`이 없으면 OAuth redirect origin 계산이 forwarded header로 fallback하지 않고 `PRODUCTION_APP_ORIGIN_REQUIRED`로 실패하게 했어.
- OAuth fail-closed 회귀 테스트와 Next 패치 버전 회귀 테스트를 추가했어.
