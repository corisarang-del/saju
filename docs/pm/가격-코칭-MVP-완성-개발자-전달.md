# 가격 코칭 MVP 완성 개발자 전달

## 배경
- 현재 제품은 사주 채팅 MVP와 첫 상담 품질은 많이 올라왔다.
- 다만 유료 전환 가능한 코칭형 사주앱으로 보려면 가격 정책과 코칭 루프가 아직 덜 닫혔다.
- 이 문서는 개발자가 바로 구현할 수 있도록 가격, 결제, 코칭 스냅샷, 오늘피드, 월간 리포트, 테스트 기준을 결정 완료 상태로 정리한 전달 문서다.

## 최종 목표
- 현재 “사주 채팅 MVP”를 “유료 전환 가능한 코칭형 사주앱 MVP”로 확장한다.
- 가격은 기존 별 패키지에 스타터 상품과 월간 멤버십을 추가한다.
- 코칭은 첫 상담 답변에서 끝나지 않고 `첫 상담 -> 코칭 요약 -> 오늘피드 -> 후속 질문 -> 월간 전략 리포트 -> 기억 요약`으로 이어지게 한다.
- 보안/결제/빈 응답 재검증은 기존 테스터 release gate로 계속 유지한다.

## 확정 가격 정책

### 무료/차감 기준
- 가입 후 무료: `3회 무료`
- 채팅: `1별 = 메시지 1회`
- 월간 전략 리포트 상세판: `3별`
- 종합 사주 백서: `5별`

### 별 패키지
- 스타터: `10별 3,900원`
- 기본: `30별 9,900원`
- 인기: `70별 19,900원`
- 최고 가성비: `250별 39,900원`

### 멤버십
- 상품명: `월간 멤버십`
- 가격: `월 9,900원`
- 혜택: 매월 `40별` 지급
- 부가 혜택: 오늘피드/월간 리포트 우선 노출
- 사용 방식: 지급된 별은 일반 별과 동일하게 채팅/리포트/백서에 차감한다.

## 가격 정책 구현 요구사항
- 가격 정책 source of truth를 하나로 통합한다.
- 랜딩, 코인샵, FAQ, 약관, 환불정책, JSON-LD, Paddle product/price 매핑은 모두 같은 가격 정책을 참조해야 한다.
- Paddle 상품 타입은 다음으로 고정한다.
  - `stars10`
  - `stars30`
  - `stars70`
  - `starsPremium`
  - `monthlyMembership`
- 결제 웹훅은 실제 `items.price.id` 기준으로만 별 지급 또는 멤버십 활성화를 처리한다.
- `customData.productType`은 사용자 매핑 등 보조 정보로만 쓰고 상품/지급량의 신뢰 근거로 쓰지 않는다.
- 클라이언트가 보낸 amount나 productType으로 별 차감/충전량이 바뀌면 안 된다.

## 코칭 스냅샷 모델

### 모델명
- `CoachingSnapshot`

### 필드
- `readingId`
- `concern`
- `todayDo`
- `todayAvoid`
- `relationshipTip`
- `followUpQuestion`
- `weeklyFocus`
- `monthlyFocus`
- `sourceMessageId`
- `createdAt`

### 생성 규칙
- 첫 상담 assistant 응답 저장이 성공한 뒤 `CoachingSnapshot`을 생성한다.
- 첫 상담 실패, 빈 응답, 부분 응답, provider error 상태에서는 snapshot을 만들지 않는다.
- snapshot 생성 실패는 채팅 응답 자체를 실패로 돌리지 않는다.
- snapshot 생성 실패는 서버 로그에 `readingId`, `sourceMessageId`, `reason`을 남긴다.
- 오늘피드와 리포트는 snapshot이 있으면 우선 사용하고, 없으면 최신 reading과 memory 기반 fallback을 사용한다.

## 대표 고민별 코칭 템플릿

### 썸/재회
- 핵심 축: 연락 패턴, 감정 소모, 기다릴지 말지 기준
- `todayDo`: 상대 반응보다 내 감정 소모 지점을 먼저 정리
- `todayAvoid`: 확답을 재촉하거나 불안해서 장문 보내기
- `relationshipTip`: 마지막 연락 흐름을 짧게 복기
- `followUpQuestion`: 최근 연락에서 가장 마음이 흔들린 순간은 언제였어?

### 이직/퇴사
- 핵심 축: 버틸 조건, 움직일 조건, 준비할 항목
- `todayDo`: 지금 직장에서 버틸 이유와 떠날 이유를 각각 3개씩 적기
- `todayAvoid`: 감정이 올라온 날 바로 퇴사 결정하기
- `relationshipTip`: 상사/동료와 말해야 할 기준을 정리
- `followUpQuestion`: 지금 제일 힘든 건 사람, 돈, 성장 중 뭐에 가까워?

### 돈 모으기
- 핵심 축: 새는 돈, 늘릴 수입, 이번 달 지출 기준
- `todayDo`: 이번 달 고정 지출과 충동 지출을 분리
- `todayAvoid`: 불안해서 갑자기 큰 투자나 소비 결정하기
- `relationshipTip`: 돈 이야기를 비교나 자책으로 끌고 가지 않기
- `followUpQuestion`: 요즘 제일 자주 새는 돈은 식비, 쇼핑, 구독 중 어디야?

### 번아웃
- 핵심 축: 줄일 일, 회복 루틴, 다시 시작할 작은 행동
- `todayDo`: 오늘 반드시 하지 않아도 되는 일을 하나 덜어내기
- `todayAvoid`: 쉬는 시간을 또 성과로 만들기
- `relationshipTip`: 도움 요청을 미루지 않기
- `followUpQuestion`: 지금 제일 먼저 줄이고 싶은 건 일, 사람, 생각 중 뭐야?

### 친구/가족관계
- 핵심 축: 거리 조절, 말투, 경계선
- `todayDo`: 내가 감당할 수 있는 선을 한 문장으로 정리
- `todayAvoid`: 미안해서 바로 다 받아주기
- `relationshipTip`: 짧고 부드럽게 선을 말하기
- `followUpQuestion`: 지금 필요한 건 더 가까워지는 것보다 거리를 조절하는 쪽이야?

## 오늘피드 구현 요구사항
- 오늘피드는 `CoachingSnapshot`을 우선 사용한다.
- snapshot이 없으면 현재처럼 최신 reading, concerns, 최근 대화 memory 기반 fallback을 보여준다.
- 상단 한 문단은 오늘의 핵심 흐름을 보여준다.
- 행동카드 3개는 다음으로 고정한다.
  - `하면 좋은 것`: `todayDo`
  - `피할 것`: `todayAvoid`
  - `관계/말투`: `relationshipTip`
- 타임라인은 아침, 점심 이후, 저녁 3개를 유지한다.
- CTA 문구는 `이 흐름으로 더 물어보기`로 한다.
- CTA는 현재 캐릭터 채팅으로 이어지며, 가능하면 `followUpQuestion`을 추천 질문으로 노출한다.

## 월간 전략 리포트 구현 요구사항
- 월간 전략 리포트는 snapshot, 최근 대화 memory, 사주 요약을 묶어서 보여준다.
- 무료 미리보기는 전체 흐름 1문단과 섹션 제목만 보여준다.
- 상세판은 `3별` 차감 성공 후 열린다.
- 상세판 구성은 다음으로 고정한다.
  - 관계
  - 일
  - 돈
  - 마음관리
  - 조심할 시기
  - 이번 달 선택 3개
- 종합 사주 백서는 `5별` 차감 후 기존 사주 분석/월운/강약/오행/자미두수/별자리 PDF 또는 상세 결과로 연결한다.
- 별 부족 시 코인샵으로 유도하고, 실패 상태에서 상세 내용을 열면 안 된다.

## 대화 기억 요약
- 최근 user 메시지 8개와 assistant 요약을 바탕으로 유지한다.
- 요약 필드는 다음을 포함한다.
  - 반복 고민
  - 사용자가 자주 묻는 기준
  - 다음에 이어 물어볼 질문
  - 캐릭터 친근도 톤
- 후속 질문 추천은 snapshot의 `followUpQuestion`과 최근 기억 요약을 함께 사용한다.

## 관리자/운영 화면 요구사항
- 관리자 화면에는 다음 정보를 표시한다.
  - 사용자 별 잔액
  - 멤버십 상태
  - 최근 차감 타입
  - 최근 snapshot 생성 여부
- 별 수동 조정은 기존 보안 기준대로 감사 로그를 유지해야 한다.
- 가격 변경 시 수정해야 하는 위치가 하나인지 테스트로 보장한다.

## 테스트 계획

### 가격 정책 테스트
- `publishes_starter_existing_and_subscription_products`
- `uses_same_price_policy_for_landing_terms_jsonld_and_paddle_mapping`
- `charges_one_star_for_chat_three_for_monthly_report_five_for_full_report`

### 결제/보안 테스트
- 싼 상품 결제에 다른 `customData`가 들어와도 실제 Paddle price 기준으로만 별 지급
- 멤버십 결제는 40별 지급과 subscription 상태 기록
- 클라이언트 amount 조작으로 리포트/백서 차감량 변경 불가
- 별 부족 상태에서 월간 리포트 상세판과 종합 사주 백서가 열리지 않음

### 코칭 테스트
- 첫 상담 성공 후 `CoachingSnapshot` 생성
- 첫 상담 실패/빈 응답/부분 응답에서는 snapshot, 별 차감, `chat_used` 증가 없음
- 대표 고민 5개가 각각 올바른 코칭 템플릿 필드를 만든다
- 오늘피드가 snapshot이 있으면 snapshot 내용을 우선 표시하고, 없으면 fallback 표시
- 월간 리포트 상세판은 3별 차감 성공 후에만 열린다

### QA 시나리오
- 신규 가입 -> 3회 무료 -> 첫 상담 -> 오늘피드 반영 -> 후속 질문 -> 월간 리포트 미리보기 -> 3별 차감 상세 보기
- 별 부족 상태에서 월간 리포트 상세 보기 시 코인샵 유도
- 멤버십 결제 후 별 40개 지급, 채팅/리포트에서 동일 별 잔액 차감
- 대표 고민 5개 live QA: `썸/재회`, `이직/퇴사`, `돈 모으기`, `번아웃`, `친구/가족관계`

## 릴리즈 게이트
- 기존 테스터 재검증 범위인 보안, 결제, 빈 응답, 첫 상담 품질은 별도 release gate로 계속 유지한다.
- 결제 포함 공개 배포 전에는 Paddle production env, 운영 Supabase migration, 별 차감 동시성, 결제 상태 전이 보안을 다시 확인한다.
- 가격/코칭 MVP는 위 release gate가 깨지지 않는 범위에서만 병합한다.

## 개발 원칙
- 개발자는 TDD 방식으로 실패 테스트를 먼저 작성한다.
- 외부 AI, DB, 결제, 시간은 시스템 경계로 보고 테스트에서는 fake 처리한다.
- 테스트는 구현 세부가 아니라 관찰 가능한 제품 동작을 검증한다.
- Codex는 PM 역할이므로 이 문서는 구현 지시서이고, 실제 코드는 개발자가 작성한다.
