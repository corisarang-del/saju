# 보안리뷰 핫픽스 수정후 재리뷰

## 요청
- 서버/보안 담당자 관점에서 개발자 보안 핫픽스 수정본을 다시 리뷰했다.
- 소스코드는 수정하지 않았다.
- 이전 보안 리뷰 내용, 이번 재리뷰 결과, 검증 결과를 문서화하고 메모리에 저장했다.

## 확인한 개선
- `saju_readings`, `saju_chat_messages`, `saju_compatibilities` RLS가 다시 enable 상태로 바뀌었다.
- `preview`, `analyze`, `compatibility`, `chat`, `pdf`, `reading/[id]` 경로에 인증 유저와 `user_id` 소유자 필터가 추가됐다.
- `services/saju/actions.ts`, `services/saju/chat-actions.ts`의 주요 Server Action에도 소유자 검증이 들어갔다.
- `/api/saju/deduct-stars`는 클라이언트 `amount`를 더 이상 사용하지 않고 `REPORT_STAR_COST`와 원자적 RPC를 사용한다.
- Paddle 웹훅은 브라우저 `customData.productType` 대신 실제 결제 `items.price.id`로 상품을 해석하고, `paddle_transaction_id` unique index와 idempotency 처리를 추가했다.
- P0 핫픽스 회귀 테스트가 추가됐다.

## 남은 릴리즈 차단 이슈

### P0: 별 충전 RPC가 authenticated에 열려 있음
- 위치: `supabase/migrations/202607010010_p0_security_hotfix.sql`
- `credit_stars_for_paddle_purchase`는 `security definer` 함수이고 `p_credits`, `p_user_id`, `p_transaction_id`를 인자로 받는다.
- 그런데 `grant execute on function public.credit_stars_for_paddle_purchase(...) to authenticated, service_role;`가 있어 로그인 유저가 Supabase RPC로 직접 호출할 수 있다.
- 이 상태면 결제 웹훅을 거치지 않고 임의 transaction id와 credits로 본인 또는 타인에게 별을 충전할 수 있다.
- 조치: 해당 함수는 `authenticated` grant를 제거하고 `service_role`만 실행하게 해야 한다. 웹훅 라우트처럼 서버 전용 service role 경로에서만 호출해야 한다.

### P0: 별 차감/기존 decrement RPC도 authenticated에 열려 있음
- 위치: `supabase/migrations/202606260050_star_credit_schema.sql`, `supabase/migrations/202607010010_p0_security_hotfix.sql`
- `decrement_star(p_user_id)`와 `deduct_stars_for_report(p_user_id, p_reading_id)`가 `security definer`인데 authenticated 실행 권한이 있다.
- 직접 이득보다는 타인 UUID를 알 때 별 차감 DoS 또는 거래 이력 오염이 가능하다.
- 조치: 직접 클라이언트 호출이 필요한 함수라면 함수 내부에서 `auth.uid() = p_user_id`와 reading 소유권을 강제하고, 서버 전용이면 `authenticated` grant를 제거해야 한다.

### P0: `/api/saju/update-status`로 결제 상태 우회 가능
- 위치: `src/app/api/saju/update-status/route.ts`
- 이 API는 인증과 소유자 검증은 하지만 클라이언트가 보낸 `status`를 그대로 저장한다.
- 사용자가 본인 reading을 직접 `paid`로 바꾼 뒤 `/api/saju/analyze`를 호출하면 유료 분석 조건을 통과할 수 있다.
- 조치: 일반 클라이언트가 임의 status를 변경하지 못하게 하고, 결제/별 차감이 성공한 서버 내부 경로에서만 `paid` 전이를 허용해야 한다. 허용 전이도 enum과 현재 상태 기반으로 제한해야 한다.

## 남은 중요 이슈

### P1: 인증 없는 Gemini suggestions API
- 위치: `src/app/api/saju/suggestions/route.ts`
- 인증, rate limit, daily quota 없이 Gemini `generateText`를 호출한다.
- 비용 폭탄과 abuse 경로가 남아 있다.
- 조치: 인증 또는 IP rate limit, user/IP daily quota, 입력 길이 제한, 429 응답을 추가해야 한다.

### P1: 채팅 AI 호출의 rate limit/daily quota 부족
- 위치: `src/app/api/saju/chat/route.ts`
- 별 잔액 체크는 있지만 분당 제한과 일일 제한은 보이지 않는다.
- 동시 요청으로 스트림 여러 개가 시작되면 비용과 별 차감 순서가 흔들릴 수 있다.
- 조치: userId + IP 기준 rate limit, active generation lock, daily quota/audit log를 추가하는 게 좋다.

### P2: PostCSS moderate 취약점 유지
- `pnpm audit --prod` 결과 `postcss <8.5.10` moderate 1건이 남아 있다.
- 경로: `next@16.2.9 > postcss@8.4.31`, `next-intl@4.13.0 > next@16.2.9 > postcss@8.4.31`
- 조치: Next/PostCSS 패치 가능 버전 확인 후 pnpm 기준으로 업데이트해야 한다.

## 검증 결과
- `pnpm audit --prod`: 실패. PostCSS moderate 1건.
- `pnpm test`: 실패. 출력 없이 오래 실행 후 `[ERROR] fetch failed`.
- 보안 관련 테스트만 따로 실행: 실패. 동일하게 `[ERROR] fetch failed`.
- 테스트 러너 실패 원인은 이번 리뷰에서 확정하지 못했다. 네트워크 fetch 의존 또는 현재 pnpm 실행 환경 문제로 분리 확인이 필요하다.

## 결론
- 이전 P0 중 IDOR, 클라이언트 amount 신뢰, Paddle priceId 검증은 코드상 상당 부분 개선됐다.
- 하지만 `security definer` RPC를 `authenticated`에 열어둔 점과 `update-status` 결제 상태 우회가 남아 있어서 아직 공개 배포는 막아야 한다.
- 다음 수정은 반드시 TDD로 `authenticated` RPC 직접 호출 방어, `paid` 상태 임의 변경 차단, suggestions/chat rate limit을 먼저 실패 테스트로 고정한 뒤 구현해야 한다.
