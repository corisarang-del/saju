# 핵심 사주 스키마 마이그레이션 추가

## 배경

Supabase에서 `public.saju_readings`를 찾을 수 없다는 schema cache 오류가 발생했다. 이전에 실행한 스키마는 별 크레딧용 `user_stars`, `star_transactions` 중심이라, 실제 사주 분석/채팅/궁합 기능이 사용하는 핵심 테이블이 없었다.

## 작업

- `supabase/migrations/202606300010_core_saju_schema.sql` 추가.
- `saju_readings`, `saju_chat_messages`, `saju_compatibilities` 테이블 생성 SQL 작성.
- 코드가 섞어 쓰는 `saju_compatibility`는 `saju_compatibilities`를 바라보는 `security_invoker` 뷰로 제공.
- RLS 정책, `updated_at` 트리거, 선택적 `star_transactions.reading_id` FK, PostgREST schema reload 알림 추가.
- `src/lib/database/core-saju-schema.test.ts`로 핵심 테이블/컬럼 회귀 테스트 추가.

## 검증

- `./node_modules/.bin/vitest run src/lib/database/core-saju-schema.test.ts` 통과.
- 현재 `/opt/homebrew/bin/pnpm` 실행이 `fetch failed`로 실패해서 `pnpm test`는 직접 확인하지 못했다.

## 추가 수정

- Supabase SQL Editor에서 `relation "public.saju_chat_messages" does not exist` 오류가 발생했다.
- RLS 정책의 서브쿼리에서 현재 테이블 컬럼을 `saju_chat_messages.reading_id`, `saju_compatibilities.reading_id`처럼 한정하던 부분을 `reading_id` 직접 참조로 변경했다.
- 회귀 테스트에 테이블 한정 정책 참조 금지 케이스를 추가했다.
- `./node_modules/.bin/vitest run` 통과: 9 files / 24 tests.

## 추가 수정 2

- 같은 `relation "public.saju_chat_messages" does not exist` 오류가 반복됐다.
- `drop policy if exists ... on public.saju_chat_messages`는 정책이 없어도 대상 테이블이 없으면 실패할 수 있어, 정책 정리/생성 구간을 `to_regclass(...)`로 감싸고 동적 SQL로 실행하도록 변경했다.
- `drop_saju_policy_if_table_exists` 헬퍼 함수를 추가했다.
- 회귀 테스트에 누락 테이블 상태의 정책 정리 위험 구문 금지 케이스를 추가했다.
- `./node_modules/.bin/vitest run` 통과: 9 files / 25 tests.

## 추가 수정 3

- Supabase SQL Editor에서 `syntax error at or near "check"` 오류가 발생했다.
- 핵심 목표가 테이블 생성과 schema cache 해결이므로, 부트스트랩 스키마에서 RLS 정책 DDL을 제거했다.
- 이전 실패 실행으로 RLS만 켜져 앱 접근이 막히는 상황을 피하려고 핵심 3개 테이블은 `disable row level security`로 정리했다.
- 보안 RLS 정책은 테이블 생성 확인 후 별도 마이그레이션으로 다시 붙이는 방향으로 분리한다.
- `./node_modules/.bin/vitest run` 통과: 9 files / 25 tests.
