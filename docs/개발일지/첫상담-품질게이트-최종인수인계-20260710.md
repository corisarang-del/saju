# 첫상담 품질게이트 최종인수인계

## 요약
- 첫 상담 실제 API 품질 불안정 문제를 수정하고 검증했다.
- 핵심은 첫 상담을 스트리밍으로 바로 보여주지 않고, 서버에서 생성 후 품질 게이트를 통과한 답변만 저장/전송하는 구조다.
- 모델 응답이 3회 연속 게이트를 통과하지 못하면 503으로 끝내지 않고, 캐릭터별 안전 fallback을 검수해 저장/전송한다.

## 수정된 핵심 파일
- `src/app/api/saju/chat/route.ts`
  - 첫 상담 비스트리밍 선검수.
  - `buildSafeInitialAnalysisFallback` 추가.
  - `usedFallback` 로그 추가.
  - `qualityReport` 구조화 로그 추가.
- `src/lib/ai/chat-completion-guard.ts`
  - 정확히 2문단, 질문 종료, 사주 근거, 두 번째 문단의 오늘 행동, 이모지/영문자/가벼운 외래어 금지를 검사.
  - `getInitialAnalysisQualityReport` 추가.
- `src/lib/saju/initial-analysis.ts`
  - 첫 상담 지시문의 `1~3문단`과 `정확히 2문단` 충돌 제거.
  - 영문자와 가벼운 외래어 금지 강화.
- `scripts/qa-gemini-first-consultation.mjs`
  - Gemini live QA 기준을 내부 게이트와 정렬.
- `scripts/qa-live-api-check.mjs`
  - 실제 API QA의 오늘 행동 검사 기준을 두 번째 문단으로 정렬.
  - 중간 `setBalance(3)` 이후 최종 잔액 기대값을 `2`로 수정.

## 운영 Supabase 조치
- 원격 Supabase migration history가 비어 있었다.
- 전체 `supabase db push`는 오래된 마이그레이션 재적용 위험이 있어 피했다.
- `supabase db query --linked --file supabase/migrations/202607070900_user_memberships_and_chat_transaction_log.sql`만 targeted 적용했다.
- 적용 후 `decrement_star`가 `star_transactions`에 `chat_message:-1` 거래 로그를 남기는 것을 확인했다.

## 다음 작업자가 주의할 점
- 첫 상담 품질은 Gemini 단독 QA만으로 완료 판단하지 않는다. 반드시 실제 `/api/saju/chat` 경로를 QA해야 한다.
- 원격 Supabase migration history가 비어 있으므로 운영 DB에는 전체 `db push`보다 targeted SQL 적용을 우선 검토한다.
- 첫 상담 fallback은 “모델 실패를 숨기는 장치”가 아니라 사용자에게 빈 응답/503 반복을 주지 않기 위한 안전망이다.
- `decrement_star`에 앱 코드에서 별도 거래 로그 insert를 추가하면 최신 RPC 적용 환경에서 이중 로그가 생길 수 있다.
